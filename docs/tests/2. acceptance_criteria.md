
# Acceptance Criteria Documentation

## Purpose

This document defines the acceptance criteria required for MVP release.

The purpose is to verify that the engineering assistant system is:

- functional
- deterministic
- traceable
- reproducible
- auditable
- explainable


The MVP acceptance philosophy:

> The MVP is successful when engineering decisions are reproducible, explainable, and auditable.

---

# 1. MVP Definition

The MVP represents a reusable engineering calculation framework.

The first fully implemented workflow is:

## ASME B31.3 Pipe Wall Thickness Verification


The workflow includes:


- user request handling
- intent identification
- graph traversal
- dependency resolution
- input collection
- validation
- deterministic calculation
- trace storage
- report generation


---

# 2. MVP Release Goal


The MVP validates architecture correctness.


The system must prove:


```

User Intent

↓

Planner

↓

Graph

↓

Validation

↓

Execution

↓

Report

````


The objective is not only obtaining a calculation result.

The system must demonstrate:

- why the calculation was selected
- which dependencies were used
- what decisions were made
- how the final result was produced


---

# 3. Supported Interface


MVP supports:


## CLI + Chat Interaction


The user can:

- start a task

- receive reports

# 8. AI Boundary Acceptance

AI must:

- manage user interaction
    
- explain results
    
- ask questions
    
- summarize reports
    

AI must not:

- calculate engineering values
    
- override deterministic results
    
- decide engineering truth
    

Principle:

> Agents decide what to do next, but never what the truth is.

# 10. Error Handling

When required information is missing:

The system must:

1. identify missing parameter for the **current navigation phase**
2. present **numbered options** when the node declares a finite choice set (`interactions.options`, yes/no assumptions)
3. explain why the value is required (node question or formula parameter description)
4. request the value without an LLM call when a deterministic prompt template applies

Example (first turn, pipe wall thickness):

```
Before expanding ASME B31.3 §304.1.1 … confirm the following assumption:

Is the pipe wall thickness … for a straight section of pipe?

Choose one:
  1. Yes — straight pipe section (supported)
  2. No — fitting or bend (not yet supported)
```

Example (path decision):

```
Choose one:
  1. Internal pressure — design per §304.1.2
  2. External pressure — design per §304.1.3
```

Example (parameter phase, after expansion gate):

```
Formula:
  t = PD / (2(SEW + PY))

Missing parameters:
  D
    Description: Outside diameter of pipe
    Needed: requires NPS for lookup …
    Choose one:
      1. Nominal pipe size (NPS) — look up …
      2. Outside diameter — enter D directly
```

# 12. Changed Inputs

If a user changes a parameter:

Example:

```
Temperature:

400F → 600F
```

The system must:

- detect affected calculations
    
- identify dependent nodes
    
- create a new calculation branch
    
- preserve previous results
    

Example:

```
Calculation A

Temperature = 400F


Calculation B

Temperature = 600F
```

# 24. Testing Threshold

## Happy Path

Example:

Example:

Example:

Change input → rebuild affected workflow

## Edge Cases

Example:

Overrides, conflicts, limitations

---

# 25. Final MVP Acceptance Checklist

MVP is accepted when:

✓ User request reaches Planner

✓ Planner creates correct task

✓ Graph finds required nodes

✓ Dependencies are resolved

✓ Validation catches problems

✓ Execution produces deterministic results

✓ Trace is stored

✓ Reports are generated

✓ State can resume tasks

✓ Changed inputs create new branches

✓ Results are reproducible

---

# Final Principle

The system is not considered complete because it calculates a number.

It is complete when it can explain:

- why the calculation happened
    
- what information was used
    
- what rules were applied
    
- what decisions were made
    
- how the final result can be reproduced
