# Developer Inspection Framework
Version: 1.0

## Goal

Implement a complete developer-only debugging environment that allows verification that the application is executing entirely through the Knowledge Graph.

This functionality must NEVER be exposed in Production mode.
I have already implemented some features, detect these first then add what is missing.
This is a big task, execute this document one phase at a time.

---

# Global Requirements

- Everything described below is DEV ONLY.
- Guard all features behind `DEV_MODE`.
- Production builds should tree-shake or completely disable these features.
- None of these tools should affect runtime behaviour.
- They are read-only inspection tools.

---

# Phase 1 — Execution Trace

## Goal

Record every node that the planner executes.

This becomes the equivalent of a stack trace for the graph.

---

## Requirements

Create an Execution Trace panel.

Each execution step should record:

- Timestamp
- Workflow ID
- Current Node ID
- Node Type
- Incoming Edge
- Outgoing Edge
- Why this node was selected
- Inputs consumed
- Outputs produced
- Execution duration
- Status
    - Success
    - Failed
    - Skipped

Example:

Step 17

Node:
shell_required_thickness

Reason:
Dependency satisfied

Inputs:
pressure
diameter
stress

Outputs:
required_thickness = 18.2 mm

Duration:
3 ms

---

## Acceptance Criteria

✓ Every executed node appears exactly once.

✓ Execution order matches planner traversal.

✓ Failed nodes remain visible.

✓ Planner decisions are explainable.

---

# Phase 2 — Value Provenance

## Goal

Every value shown anywhere in the application must be traceable back to the originating node.

Nothing should appear "magically."

---

## Requirements

Every displayed value should expose its provenance.

Applicable to:

- Parameters
- Equations
- Table cells
- Figures
- Text paragraphs
- Warnings
- Assumptions
- Reports

Developer should be able to inspect:

Source Node

Source Property

Transformation history

Dependencies

Consumers

Example

Displayed Value

138 MPa

Developer View

Source Node:
allowable_stress

Property:
outputs.value

Generated by:
material_lookup

Consumed by:
shell_formula

---

## Acceptance Criteria

✓ Every displayed value has a source.

✓ Missing provenance produces a warning.

✓ Zero hardcoded UI values.

# Phase 5 — Graph Integrity Tests

## Goal

Detect accidental hardcoding.

---

## Test 1

Rename node.

Example

shell_formula

↓

banana_formula

Application should continue working after references update.

No TypeScript should depend on filenames.

---

## Test 2

Rename display title.

Application should update automatically.

No UI labels hardcoded.

---

## Test 3

Move node to another folder.

Application should still resolve graph.

No folder assumptions.

---

## Test 4

Disable node.

Planner should fail gracefully.

Developer should see

Missing dependency

instead of silent fallback.

---

## Acceptance Criteria

✓ Graph is source of truth.

✓ No filename coupling.

✓ No folder coupling.

✓ No hidden shortcuts.

# Definition of Done

The application is considered graph-driven when:

✓ Every UI value has provenance.

✓ Every planner decision is explainable.

✓ Every execution step is recorded.

✓ Every traversal is visualized.

✓ Renaming nodes does not break execution.

✓ Disabling nodes produces dependency errors.

✓ No hidden TypeScript shortcuts exist.

✓ A developer can understand every output solely by using the Developer Inspector.
