# Purpose

This document defines the data structures used throughout the engineering knowledge graph system.

The data model separates:

- engineering truth
- validation rules
- execution state
- user interaction
- reporting
- traceability


The system follows the principle:

> Agents decide what to do next, but never what the truth is.


---

# 1. Data Model Architecture


The system consists of the following primary data domains:

```
Standard Knowledge Model
|  
v
Graph Model
|  
v
Execution Model
|  
v
Rule Validation Model
|  
v
Report Model
|  
v
Session Memory Model
```


Each model has a defined responsibility.

---

# 2. Standard Node Model

## Purpose

Represents one engineering knowledge unit.

A node may represent:

- paragraph
- formula
- lookup
- decision
- constraint
- informational requirement


A node is deterministic.

Given the same inputs:
```
same node + same inputs = same output
```


---

## Node Structure


Recommended structure:
```
standards/

	asme_b31.3/

		nodes/

			B313-304.1.1/
				node.md  
  
  
```

A node folder is preferred because one paragraph may contain:  
  
- multiple formulas  
- multiple conditions  
- multiple notes  
- exceptions  
- references  
  
  
---  
  
# Node Metadata Schema  
  
  
Example:  
  
  
```yaml  
id:  
  
	B313-304.1.1  
    
  
paragraph:  
  
	304.1.1  
  
  
text:  
  
	the actual text of paragraph

formula:
	t = PD / 2(SE + PY)

notes: 
	
references:

assumptions:

limitations:

exceptions:
  
```

# Node Content

Every node contains:

## Standard Text

The exact paragraph text.

This provides:

- traceability
- audit capability
- reference integrity

---

## Assumptions

Example:
```
assumptions:  
  
- material properties are valid for design temperature  
  
- pipe geometry is circular
```
---
## Limitations

Example:
```
limitations:  
  
- not applicable above temperature limit  
  
- requires thin wall condition
```
---
## Exceptions

Example:
```
exceptions:  
  
- does not apply to external pressure
```
## Notes

a paragraph may have multiple notes.

Example:

```
notes:  
  
- corrosion allowance must be considered separately
```

# Formula Model

A node may contain multiple formulas.

Each formula is stored separately.

Example:

```
formulas/

	thickness.md

	pressure_check.md

```

Formula metadata:

```
formula_id:

	wall_thickness


name:

	Internal Pressure Thickness


display:

	t = PD / 2(SEW+PY)


steps:


	- calculate SEW
	  
	- calculate PY
	
	- calculate PD

	- calculate required thickness

	- apply allowances

```

# Calculation Step Model

Every calculation stores intermediate values.

Example:

```yaml
step:

name:

	SEW calculation


inputs:

S:
	17500 psi

E:
	1.0

W:
	1.0


result:

	17500 psi
```

Intermediate values are not rounded.

Rounding occurs only during final presentation.

input values should be first verified for consistency in their units (SI unit of measurement should be the default).

# 3. Graph Model

## Purpose

Represents the dependency relationship between nodes.

The graph is versioned.

---

# Graph Version

Example:
```yaml
graph_id:

	pipe_integrity_check

	timestamp


standard_versions:

	ASME B31.3 2024
```

- reruns
- report rebuilding

Example:

```yaml

	B313-304.1.1

	B313-345
```

- dependency
- reference
- conditional
- informational
- validation

```yaml

	500

```

# Input Properties

Inputs support:

## Default Values

```yaml
default:  
  
1.0  
  
  
requires_confirmation:  
  
true
```

---

## Range Validation

```yaml
limits:  
  
min:  
  
0  
  
  
max:  
  
900
```

## Uncertainty

Example:

```yaml
uncertainty:

±5%

```

## Table Lookup Rules

Tables do not interpolate.

Rule:

If exact value does not exist:

return the lower applicable table value.

Example:

```text
Requested:

	425°F

Table values:

	400°F

	500°F

Selected:

	400°F

```


# 6. Task State Model

## Purpose

Tracks active engineering tasks.

A session can contain multiple tasks.

Structure:

```
session  
  
|  
  
+-- task_001  
  
|  
  
+-- task_002
```

# Task Model

Example:

```yaml
task_id:

pipe_integrity_check


status:

in_progress


active_nodes:

[]


inputs:

{}


outputs:

{}


warnings:

[]

```

---
# Contradiction Detection

The system compares inputs between tasks.

Example:

Task 1:
```
Temperature = 300F
```
Task 2:
```
Temperature = 600F
```

The system records:
```
conflict:

previous_calculation_invalid:

true


reason:

input changed

```

The user must be informed:

"The previous calculation may no longer be valid. A new full execution is required."

# 10. AI Context Model

Agents receive controlled context.

```yaml

	pipe_design

	[]

	[]

	{}
```

---

# Agent Restrictions

Agents cannot modify:

- formulas
- values
- outputs
- decisions
- references

Agents only:

- select navigation path
- request missing information
- explain results

---

# 11. Prompt Model

Prompts are separated from code.

Structure:

```
ai/

prompts/

- intent_agent.md
- planner_agent.md
- input_agent.md
- routing_agent.md
- synthesis_agent.md
- intent_detection.md
- node_selection.md
- missing_information.md
- report_explanation.md
```

Each prompt defines one AI responsibility.

---

# Final Principle

The data model ensures:

Graph:

	defines what happens.

Rules:

	define what is allowed.

Agents:

	decide where to look.

Reports:

	explain what happened.

Every engineering result remains reproducible and traceable.
