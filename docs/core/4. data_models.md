# Purpose

This document defines the data structures used throughout the engineering knowledge graph system.

The data model separates:

- engineering truth
- validation rules
- execution state
- user interaction
- reporting
- traceability


The system follows the principle:

> Agents decide what to do next, but never what the truth is.


---

# 2. Standard Node Model

## Purpose

Represents one engineering knowledge unit.

A node may represent:

- paragraph
- formula
- lookup
- decision
- constraint
- informational requirement


A node is deterministic.

Given the same inputs:
```
same node + same inputs = same output
```


---

## Node Structure


Recommended structure:
```
standards/

	asme_b31.3/

		nodes/

			B313-304.1.1/
				node.md
				formulas/
				conditions/
				notes/
				references/
```

> **Reference implementation:** `standards/asme_b31.3/nodes/B313-304.1.1/` (see `7. node_structure_design.md` §19).

A node folder is preferred because one paragraph may contain:  
  
- multiple formulas  
- multiple conditions  
- multiple notes  
- exceptions  
- references  
  
  
---

# Formula Model

A node may contain multiple formulas.

Each formula is stored separately.

Example:

```
formulas/

	thickness.md

	pressure_check.md

```

Formula metadata:

```
formula_id:

	wall_thickness


name:

	Internal Pressure Thickness


display:

	t = PD / 2(SEW+PY)


steps:


	- calculate SEW
	  
	- calculate PY
	
	- calculate PD

	- calculate required thickness

	- apply allowances

```

# Calculation Step Model

Every calculation stores intermediate values.

Example:

```yaml
step:

name:

	SEW calculation


inputs:

S:
	17500 psi

E:
	1.0

W:
	1.0


result:

	17500 psi
```

Intermediate values are not rounded.

Rounding occurs only during final presentation.

input values should be first verified for consistency in their units (SI unit of measurement should be the default).

# 4. Input Model

## Purpose

Stores user-provided and system-provided values.

the input values should be first validated for measurement unit and consistency.

Example:
```yaml
input_id:

	design_pressure


value:

	500


unit:

	psi


source:
	
	user


status:

	confirmed

symbol:

	P

description:

	Design pressure

introduced_at_node:

	B313-304.1.1

resolved_at_node:

	B313-304.1.2

resolution_method:

	user_input

```

# Input Properties

Inputs support provenance and registry metadata:

| Field | Purpose |
|-------|---------|
| `symbol` | Engineering symbol (P, D, S, …) |
| `description` | Human-readable parameter name |
| `introduced_at_node` | Definition node where the symbol is introduced |
| `resolved_at_node` | Node where the value is obtained or applied |
| `resolution_method` | `user_input`, `table_lookup`, `equation`, `node_output`, `default_confirmed`, `system` |
| `resolution_ref` | `ResolutionRef` — table id, node id, equation id, subsection |

`ParameterDescriptor` (on `Task.parameter_registry`) holds pending symbol metadata seeded from definition-node nomenclature before values are collected. It is not a substitute for `EngineeringInput` on the task.

Inputs also support:

## Default Values

```yaml
default:  
  
1.0  
  
  
requires_confirmation:  
  
true
```

---

## Range Validation

```yaml
limits:  
  
min:  
  
0  
  
  
max:  
  
900
```

## Uncertainty

Example:

```yaml
uncertainty:

±5%

```

## Table Lookup Rules

Tables do not interpolate.

Rule:

If exact value does not exist:

return the lower applicable table value.

Example:

```text
Requested:

	425°F

Table values:

	400°F

	500°F

Selected:

	400°F

```

## Navigation plan (phased)

`NavigationPlan` (`models/planning.py`) carries phase-aware missing-field state:

```yaml
current_phase: parameter_gathering

phase_missing:
  expansion_assumptions: []
  path_decisions: []
  parameter_gathering: [design_pressure, material]
  coefficient_resolution: [joint_category]
  execution_assumptions: [corrosion_allowance]

missing_assumptions: []
missing_inputs: [design_pressure, material, design_temperature]

# User-facing text is built by engine/messaging/* from phase_missing[current_phase],
# not from questions alone. questions holds planner copy for the active phase.
questions:
  - Please provide design pressure (P).
```

`NavigationPhase` values: `expansion_assumptions`, `path_decisions`, `parameter_gathering`, `coefficient_resolution`, `execution_assumptions`, `ready`.

The orchestrator exposes **only** `phase_missing[current_phase]` as `missing_inputs` in CLI responses during input collection.

# 6. Task State Model

## Purpose

Tracks active engineering tasks.

A session can contain multiple tasks.

Structure:

```
session  
  
|  
  
+-- task_001  
  
|  
  
+-- task_002
```

# Task Model

Example:

```yaml
task_id:

pipe_integrity_check


status:

in_progress


active_nodes:

[]


inputs:

{}


outputs:

{}


warnings:

[]

parameter_registry:

  design_pressure:
    symbol: P
    description: Design pressure
    introduced_at_node: B313-304.1.1

```

`parameter_registry` is populated by the graph engine after expansion assumptions pass. It tracks symbol provenance; actual values live in `inputs`.

---
# Contradiction Detection

The system compares inputs between tasks.

Example:

Task 1:
```
Temperature = 300F
```
Task 2:
```
Temperature = 600F
```

The system records:
```
conflict:

previous_calculation_invalid:

true


reason:

input changed

```

The user must be informed:

"The previous calculation may no longer be valid. A new full execution is required."

# Agent Restrictions

Agents cannot modify:

- formulas
- values
- outputs
- decisions
- references

Agents only:

- select navigation path
- request missing information
- explain results

---

# Final Principle

The data model ensures:

Graph:

	defines what happens.

Rules:

	define what is allowed.

Agents:

	decide where to look.

Reports:

	explain what happened.

Every engineering result remains reproducible and traceable.
