## Purpose

This document defines the runtime behavior of the engineering system.

It describes how a user request becomes:

- a graph execution plan
- validated engineering calculations
- traceable reports

---

# 1. System Workflow Overview

The system is a state machine + graph execution engine.

Flow:

User Request → Agent → Task Manager → Planner → Graph Engine → ValidationEngine.validate_plan() → Executor.execute_plan() → ValidationEngine.validate_node() → Report Generator → AI Presentation

---

# 2. Workflow and Task Status Vocabulary

Use separate vocabularies — do not mix them.

## Persisted `TaskStatus` (`models/task.py`)

| Value | Meaning |
| --- | --- |
| `active` | Task in progress |
| `awaiting_input` | Paused for user input on the active frontier |
| `paused` | Explicitly paused |
| `completed` | Finished successfully |
| `invalidated` | Hard failure or invalidation |
| `in_progress` | Execution underway (maps from `ExecutionContextStatus.EXECUTING`) |

`WAITING_FOR_INPUT` is **not** a persisted task status. Use `awaiting_input`.

## Navigation phases (`NavigationPhase` in `models/planning.py`)

Planner/UI phase order — not persisted as `TaskStatus`:

`expansion_assumptions` → `path_decisions` → `parameter_gathering` → `coefficient_resolution` → `execution_assumptions` → `definition_equation_completion` → `ready`

## Orchestration labels (documentation only)

Labels such as `NEW`, `INTENT_DETECTED`, `GRAPH_DISCOVERED`, `INPUT_COLLECTION`, `VALIDATION`, `EXECUTION`, and `REPORT_GENERATION` describe orchestration steps — they are **not** `TaskStatus` enum values unless added to `models/task.py`.

## Node execution status (`NodeExecutionStatus`)

`pending`, `completed`, `error`, `awaiting_input`, `skipped`

## Events (`EventType` in `models/event.py`)

Lowercase snake_case event names, e.g. `node_discovered`, `input_requested`, `calculation_completed`, `report_generated`.

# 4. Graph Expansion Lifecycle

Graph expansion is **incremental** — not a single upfront build of the complete execution graph.

Lifecycle:

1. Select workflow.
2. `expand_workflow` with current Facts.
3. If an expansion gate or branch decision is unresolved, expose only that graph-derived requirement.
4. Collect and validate permitted input.
5. Re-expand and replan (`required_user_inputs()`, planner refresh).
6. Repeat until the active subgraph and executable work are resolved.
7. `ValidationEngine.validate_plan()` then `Executor.execute_plan()` for ready nodes.

Output at each pass (partial expansion):

- active node set on the current path
- dependency edges among active nodes
- provisional execution order for resolved work
- missing inputs / gates for the current frontier

No calculation occurs during expansion-only passes.

# 5. Input Collection Workflow

Inputs are collected in **phased** order. Expansion assumptions must be confirmed before parameter gathering begins.

## Phases (pipe wall thickness reference workflow)

| Phase | Purpose | Example fields |
|-------|---------|----------------|
| 1. Expansion assumptions | Gate path expansion | `straight_pipe_section` |
| 2. Path decisions | Select internal vs external branch | `pressure_design_case` |
| 3. Parameter gathering | User-provided design values (graph-active only) | e.g. `design_pressure`, NPS/D, `material_grade`, `design_temperature` — derived from `required_user_inputs()` on the expanded path |
| 4. Coefficient resolution | Collect required **lookup inputs**; resolve E, W, Y, S automatically at runtime | e.g. `joint_category`, `pipe_construction_type`, `material_grade`, `design_temperature` — **not** direct asks for lookup outputs such as `weld_joint_efficiency`, `weld_joint_strength_reduction_factor_W`, `temperature_coefficient_Y`, or `allowable_stress` |
| 5. Execution assumptions | Post-calculation inputs | `corrosion_allowance` |

The Planner exposes one active phase at a time via `NavigationPlan.current_phase` and `phase_missing`. The CLI and extractor only accept fields allowed in the current phase.

## Deterministic user prompts

The orchestrator formats replies without LLM calls when the active phase has a known prompt template. **Prompt copy is owned by `engine/messaging/`, not the planner** (`docs/rules.md` §12).

1. **Parameter input prompt** (`engine/messaging/parameter_input_prompt.py`) — resolves the active field ask from workflow interactions, PARAM-* node metadata, and equation context.
2. **Step prompt** (`engine/messaging/step_prompt.py`) — expansion assumptions, path decisions, and other graph-defined gate or branch choices. Presents **numbered options** when `interactions.options` or yes/no gates apply. Does **not** ask users to confirm lookup-derived outputs (E, W, Y, S, allowable stress, etc.) — those resolve automatically once required lookup inputs are available (`docs/rules.md` §16).
3. **Formula parameter prompt** (`engine/messaging/formula_parameter_prompt.py`) — after the expansion gate, shows the governing equation plus **known** and **missing** parameters, each on separate lines (symbol, description, value/guidance). Unresolved lookup outputs show a standards reference link (§15), not "awaiting user input". Missing fields with finite choices include numbered options (e.g. NPS vs direct OD).
4. **Fallback** (`ai/response/response_handler.format_input_requests`) — plain bullet list only when no template applies.

User replies are parsed by `ai/input_extractor.py` (symbol labels, option numbers, aliases, confirm/yes).

## Flow Guidance Layer (traversal narration)

Traversal narration — why the workflow is at a given step, why a node is active, what happens next — is **not** parameter prompt copy. It lives in the Flow Guidance Layer (`docs/rules.md` §21).

| Component | Role |
| --- | --- |
| `GuidanceResolver` | Loads `presentation/guidance/workflows/*.yaml`; returns `GuidanceBlock` objects keyed by phase, node, traversal event, edge reason |
| `ResponseComposer` | Merges guidance blocks + deterministic messaging prompts + warnings + asset references into `PresentationResponse` |
| `models/presentation.py` | `GuidanceBlock`, `PresentationBlock`, `PresentationResponse`; `presentation_blocks` owned by `PresentationResponse` (not workflow state); `transcript_blocks` append-only history |

Deterministic parameter and step prompts remain in `engine/messaging/` (§12): `step_prompt.py`, `formula_parameter_prompt.py`, and desktop/API `parameter_input_prompt.py`. ResponseComposer combines them with guidance; it does not replace them.

`PlannerTraversalState` may supply `traversal_event` and `active_node_id` as context. Its `message` fields are debug-only and must not become user-facing guidance.

## Rules

- shared inputs collected once
- expansion assumptions confirmed before `required_user_inputs()` returns parameter fields
- **input acceptance** (parsing, type, unit validity) is enforced when values are stored where implemented; **plan-level** and **per-node** compliance run via `ValidationEngine` at execution boundaries
- missing fields in the active phase pause workflow (`TaskStatus.AWAITING_INPUT` / `awaiting_input`)
- a parameter registry is seeded from **active compiled `PARAM-*` nodes** on the expanded graph path once the expansion gate passes (legacy definition-node nomenclature seeding applies only when legacy traversal or no graph cache)
- **graph-driven paths** — which nodes and parameters are active comes from graph expansion (`expand_workflow`, edge `when`, node `applicability`); workflow nested `runtime.navigation` supplies phase **order** only, not a static list of all branch parameters. See `docs/rules.md` §13.

## Example (internal pressure, after expansion gate)

Required next:

- Design pressure (P)
- Nominal pipe size or outside diameter (D)
- Material grade and design temperature (lookup inputs for allowable stress S — S is derived, not confirmed by the user)
- Joint category or pipe construction type (lookup inputs before E/W resolution — coefficients are derived, not confirmed)
- Corrosion allowance (c, for minimum ordered thickness t_m)

# 6. Conditional Execution Workflow

Branch activation during **graph expansion** evaluates edge `when`, node `applicability`, and expansion gates via `engine/graph/conditions.py` (`when_clause_matches`) — not via independent duplicate evaluation in multiple components.

`RuleEngine` evaluates only narrow runtime expression conditions delegated during node execution (e.g. validation-rule runner). `ValidationEngine` is the compliance gate. The Executor does not select branches.

If a branch is inactive:

- branch nodes are excluded from the active subgraph
- dependent work is marked inactive or skipped
- user is informed via deterministic messaging / Flow Guidance

---

# 7. Validation Workflow

`ValidationEngine` is the compliance gate. It checks:

- unit consistency
- temperature limits
- material constraints
- table applicability
- plan readiness before execution

`RuleEngine` (`engine/rules/rule_engine.py`) is a narrow condition/rule helper for node-level edge evaluation — not the full compliance layer.

If failure:

- execution stops for affected branch
- user is asked for correction or override

---

# 8. Calculation Workflow

Calculation Engine:

- executes formulas
- preserves intermediate values
- validates units before execution
- does NOT round intermediate results

# 9. Parallel Execution

Independent graph branches may execute in parallel.

# 10. Recalculation Workflow

If input changes:

System:

1. identifies affected nodes
2. invalidates dependent results
3. keeps unaffected nodes
4. requests re-execution only where needed

# 12. Task Resume Workflow

On session start:

System checks for incomplete tasks.

User is asked:

"Do you want to continue previous analysis?"

- restore state
- resume graph execution

# 13. Override Workflow

Only findings explicitly classified as **overrideable** may be acknowledged and continued with.

| Outcome | Behavior |
| --- | --- |
| `FAIL` / non-overrideable constraint | Stop or invalidate affected execution — user confirmation must **not** silently convert to a warning |
| Overrideable finding | May proceed after explicit user decision |

An override must record: rule id, severity, reason, user identity/decision, timestamp, and affected results. Overrides are stored (e.g. `task.outputs["validation_overrides"]`) and appear in reports.

**Implementation gap:** CLI override UI is deferred; not all override paths enforce full audit metadata at input time.

---

# 14. Error Handling Workflow

On failure:

- stop affected branch
- preserve state
- ask user for correction
- allow continuation of independent nodes

# 15. Report Generation Workflow

Report pipeline (Flow Guidance is **not** included — live workflow UI only):

```
Task state + execution/validation/graph traces + referenced knowledge
↓
ReportData builder (engine/reports/report_data.py)
↓
Deterministic report generator (templates / formatters)
↓
Optional AI presentation (wording only)
↓
PDF / HTML output
```

Reports include:

- decisions
- calculations
- dependencies
- warnings
- traceability
