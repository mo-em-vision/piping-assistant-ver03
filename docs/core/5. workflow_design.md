
## Purpose

This document defines the runtime behavior of the engineering system.

It describes how a user request becomes:

- a graph execution plan
- validated engineering calculations
- traceable reports

---

# 1. System Workflow Overview

The system is a state machine + graph execution engine.

Flow:

User Request → Agent → Task Manager → Graph Engine → Rule Engine → Calculation Engine → Report Generator → AI Presentation

---

GRAPH_DISCOVERED

# 3. Intent Handling Workflow

If user intent is ambiguous:

System generates candidate tasks:

- thickness design
- pressure test
- flexibility analysis

Agent must request clarification.

No execution begins before selection.

---

# 4. Graph Discovery Phase

Graph Engine builds full dependency graph BEFORE execution.

Output:

- required nodes
- dependency edges
- execution order
- missing inputs list

No calculation occurs here.

---

# 5. Input Collection Workflow

Inputs are collected in **phased** order. Expansion assumptions must be confirmed before parameter gathering begins.

## Phases (pipe wall thickness reference workflow)

| Phase | Purpose | Example fields |
|-------|---------|----------------|
| 1. Expansion assumptions | Gate path expansion | `straight_pipe_section` |
| 2. Path decisions | Select internal vs external branch | `pressure_loading` |
| 3. Parameter gathering | User-provided design values | `design_pressure`, NPS/D, `material`, `design_temperature` |
| 4. Coefficient resolution | Confirm or resolve E, W, Y | `joint_category`, `weld_joint_efficiency`, `weld_strength_reduction`, `temperature_coefficient` |
| 5. Execution assumptions | Post-calculation inputs | `corrosion_allowance` |

The Planner exposes one active phase at a time via `NavigationPlan.current_phase` and `phase_missing`. The CLI and extractor only accept fields allowed in the current phase.

## Deterministic user prompts

The orchestrator formats replies without LLM calls when the active phase has a known prompt template:

1. **Step prompt** (`engine/messaging/step_prompt.py`) — expansion assumptions, path decisions, joint category, and coefficient confirmations. Presents **numbered options** when `interactions.options` or yes/no gates apply.
2. **Formula parameter prompt** (`engine/messaging/formula_parameter_prompt.py`) — after the expansion gate, shows the governing equation plus **known** and **missing** parameters, each on separate lines (symbol, description, value/guidance). Missing fields with finite choices include numbered options (e.g. NPS vs direct OD, confirm default E vs override).
3. **Fallback** (`ai/response/response_handler.format_input_requests`) — plain bullet list only when no template applies.

User replies are parsed by `ai/input_extractor.py` (symbol labels, option numbers, aliases, confirm/yes).

## Rules

- shared inputs collected once
- expansion assumptions confirmed before `required_user_inputs()` returns parameter fields
- validation occurs immediately when values are stored
- missing fields in the active phase pause workflow (`AWAITING_INPUT`)
- a parameter registry is seeded from definition-node nomenclature (e.g. `B313-304.1.1`) once the expansion gate passes

## Example (internal pressure, after expansion gate)

Required next:

- Design pressure (P)
- Nominal pipe size or outside diameter (D)
- Material and design temperature (for allowable stress S)
- Joint category (before E lookup)
- Corrosion allowance (c, for minimum ordered thickness t_m)

---

# 6. Conditional Execution Workflow

Conditions are evaluated early.

If condition fails:

- branch is skipped
- user is informed
- dependent nodes are marked inactive

---

# 7. Rule Validation Workflow

Rule Engine checks:

- unit consistency
- temperature limits
- material constraints
- table applicability

If failure:

- execution stops for affected branch
- user is asked for correction or override

---

# 8. Calculation Workflow

Calculation Engine:

- executes formulas
- preserves intermediate values
- validates units before execution
- does NOT round intermediate results

---

# 9. Parallel Execution

Independent graph branches may execute in parallel.

Example:

- material stress
- thickness
- pressure rating

---

# 10. Recalculation Workflow

If input changes:

System:

1. identifies affected nodes
2. invalidates dependent results
3. keeps unaffected nodes
4. requests re-execution only where needed

---

# 11. Multi-Standard Workflow

If multiple standards apply:

System presents options:

- ASME B31.3
- API 570
- BPVC VIII

User selects one before execution continues.

---

# 12. Task Resume Workflow

On session start:

System checks for incomplete tasks.

User is asked:

"Do you want to continue previous analysis?"

If yes:

- restore state
- resume graph execution

---

# 13. Override Workflow

If rule violation occurs:

User may override.

Override is:

- logged
- flagged in report
- notifies safety warning

Execution continues with warning.

---

# 14. Error Handling Workflow

On failure:

- stop affected branch
- preserve state
- ask user for correction
- allow continuation of independent nodes

---

# 15. Report Generation Workflow

Flow:

Execution Trace
↓
Report Data Model
↓
AI Presentation Layer
↓
PDF / HTML Output

Reports include:

- decisions
- calculations
- dependencies
- warnings
- traceability

---

# 16. Event Logging Workflow

System logs:

- node discovery
- input collection
- rule validation
- calculation execution
- overrides
- report generation

Supports replay and audit.

---

# 17. Final Principle

Graph defines truth.

Rules define validity.

Agents define navigation.

Everything is deterministic after execution begins.
