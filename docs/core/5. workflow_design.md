
## Purpose

This document defines the runtime behavior of the engineering system.

It describes how a user request becomes:

- a graph execution plan
- validated engineering calculations
- traceable reports

---

# 1. System Workflow Overview

The system is a state machine + graph execution engine.

Flow:

User Request → Agent → Task Manager → Planner → Graph Engine → ValidationEngine.validate_plan() → Executor.execute_plan() → ValidationEngine.validate_node() → Report Generator → AI Presentation

# 3. Intent Handling Workflow

If user intent is ambiguous:

System generates candidate tasks:

- thickness design
- pressure test
- flexibility analysis

Agent must request clarification.

No execution begins before selection.

---

# 4. Graph Discovery Phase

Graph Engine builds full dependency graph BEFORE execution.

Output:

- required nodes
- dependency edges
- execution order
- missing inputs list

No calculation occurs here.

# 5. Input Collection Workflow

Inputs are collected in **phased** order. Expansion assumptions must be confirmed before parameter gathering begins.

## Phases (pipe wall thickness reference workflow)

| Phase | Purpose | Example fields |
|-------|---------|----------------|
| 1. Expansion assumptions | Gate path expansion | `straight_pipe_section` |
| 2. Path decisions | Select internal vs external branch | `pressure_loading` |
| 3. Parameter gathering | User-provided design values (graph-active only) | e.g. `design_pressure`, NPS/D, `material_grade`, `design_temperature` — derived from `required_user_inputs()` on the expanded path |
| 4. Coefficient resolution | Collect lookup **keys**; resolve E, W, Y, S at runtime | e.g. `joint_category`, `pipe_construction_type` — **not** direct asks for `weld_joint_efficiency`, `weld_joint_strength_reduction_factor_W`, `temperature_coefficient_Y`, or `allowable_stress` unless explicitly authored as user-confirmed defaults on the node |
| 5. Execution assumptions | Post-calculation inputs | `corrosion_allowance` |

The Planner exposes one active phase at a time via `NavigationPlan.current_phase` and `phase_missing`. The CLI and extractor only accept fields allowed in the current phase.

## Deterministic user prompts

The orchestrator formats replies without LLM calls when the active phase has a known prompt template. **Prompt copy is owned by `engine/messaging/`, not the planner** (`docs/rules.md` §12).

1. **Parameter input prompt** (`engine/messaging/parameter_input_prompt.py`) — resolves the active field ask from workflow interactions, PARAM-* node metadata, and equation context.
2. **Step prompt** (`engine/messaging/step_prompt.py`) — expansion assumptions, path decisions, joint category, and coefficient confirmations. Presents **numbered options** when `interactions.options` or yes/no gates apply.
3. **Formula parameter prompt** (`engine/messaging/formula_parameter_prompt.py`) — after the expansion gate, shows the governing equation plus **known** and **missing** parameters, each on separate lines (symbol, description, value/guidance). Unresolved lookup outputs show a standards reference link (§15), not "awaiting user input". Missing fields with finite choices include numbered options (e.g. NPS vs direct OD).
4. **Fallback** (`ai/response/response_handler.format_input_requests`) — plain bullet list only when no template applies.

User replies are parsed by `ai/input_extractor.py` (symbol labels, option numbers, aliases, confirm/yes).

## Flow Guidance Layer (traversal narration)

Traversal narration — why the workflow is at a given step, why a node is active, what happens next — is **not** parameter prompt copy. It lives in the Flow Guidance Layer (`docs/rules.md` §21).

| Component | Role |
| --- | --- |
| `GuidanceResolver` | Loads `presentation/guidance/workflows/*.yaml`; returns `GuidanceBlock` objects keyed by phase, node, traversal event, edge reason |
| `ResponseComposer` | Merges guidance blocks + deterministic messaging prompts + warnings + asset references into `PresentationResponse` |
| `models/presentation.py` | `GuidanceBlock`, `PresentationBlock`, `PresentationResponse`; `presentation_blocks` owned by `PresentationResponse` (not workflow state); `transcript_blocks` append-only history |

Deterministic parameter and step prompts remain in `engine/messaging/` (§12): `step_prompt.py`, `formula_parameter_prompt.py`, and desktop/API `parameter_input_prompt.py`. ResponseComposer combines them with guidance; it does not replace them.

`PlannerTraversalState` may supply `traversal_event` and `active_node_id` as context. Its `message` fields are debug-only and must not become user-facing guidance.

## Rules

- shared inputs collected once
- expansion assumptions confirmed before `required_user_inputs()` returns parameter fields
- validation occurs immediately when values are stored
- missing fields in the active phase pause workflow (`AWAITING_INPUT`)
- a parameter registry is seeded from **active compiled `PARAM-*` nodes** on the expanded graph path once the expansion gate passes (legacy definition-node nomenclature seeding applies only when legacy traversal or no graph cache)
- **graph-driven paths** — which nodes and parameters are active comes from graph expansion (`expand_workflow`, edge `when`, node `applicability`); navigation `runtime.yaml` supplies phase **order** only, not a static list of all branch parameters. See `docs/rules.md` §13.

## Example (internal pressure, after expansion gate)

Required next:

- Design pressure (P)
- Nominal pipe size or outside diameter (D)
- Material grade and design temperature (for allowable stress S lookup keys)
- Joint category (lookup key before E/W resolution)
- Corrosion allowance (c, for minimum ordered thickness t_m)

# 6. Conditional Execution Workflow

Conditions are evaluated early.

If condition fails:

- branch is skipped
- user is informed
- dependent nodes are marked inactive

---

# 7. Validation Workflow

`ValidationEngine` is the compliance gate. It checks:

- unit consistency
- temperature limits
- material constraints
- table applicability
- plan readiness before execution

`RuleEngine` (`engine/rules/rule_engine.py`) is a narrow condition/rule helper for node-level edge evaluation — not the full compliance layer.

If failure:

- execution stops for affected branch
- user is asked for correction or override

---

# 8. Calculation Workflow

Calculation Engine:

- executes formulas
- preserves intermediate values
- validates units before execution
- does NOT round intermediate results

---

# 9. Parallel Execution

Independent graph branches may execute in parallel.

Example:

- material stress
- thickness
- pressure rating

---

# 10. Recalculation Workflow

If input changes:

System:

1. identifies affected nodes
2. invalidates dependent results
3. keeps unaffected nodes
4. requests re-execution only where needed

---

# 11. Multi-Standard Workflow

If multiple standards apply:

System presents options:

- ASME B31.3
- API 570
- BPVC VIII

User selects one before execution continues.

---

# 12. Task Resume Workflow

On session start:

System checks for incomplete tasks.

User is asked:

"Do you want to continue previous analysis?"

If yes:

- restore state
- resume graph execution

---

# 13. Override Workflow

If rule violation occurs:

User may override.

Override is:

- logged
- flagged in report
- notifies safety warning

Execution continues with warning.

---

# 14. Error Handling Workflow

On failure:

- stop affected branch
- preserve state
- ask user for correction
- allow continuation of independent nodes

---
