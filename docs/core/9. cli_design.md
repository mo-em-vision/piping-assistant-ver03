# CLI Design Documentation

## Purpose

This document defines the command line interface (CLI) design for the engineering assistant system.

The CLI is an engineering user interface that allows users to:

- interact with the AI assistant
- manage engineering tasks
- review graph traversal
- generate reports
- inspect traceability

The CLI does not contain engineering logic.

Engineering truth remains in:

- Deterministic Graph Engine
- Rule Engine
- Calculation Engine
- Report Generator


Core principle:

> The CLI is an interface layer. It does not define engineering truth.

---

# 1. CLI Architecture

The CLI communicates with the internal system components.

Architecture:
```
User

|

CLI

|

AI Interaction Layer

|

Task State Manager

|

Graph Engine

|

Rule Engine

|

Calculation Engine

|

Report Generator
```


The CLI must not bypass:

- dependency resolution
- validation
- calculation execution
- trace generation

---

# 2. Technology Selection

The CLI implementation uses:

- Python
- Typer
- Rich


## Typer

Responsible for:

- command registration
- argument handling
- validation
- help generation


## Rich

Responsible for:

- tables
- formatted output
- tree visualization
- progress display
- readable engineering reports

---

# 3. CLI User Scope

The CLI is designed for engineering users.

Primary purpose:

Allow engineers to interact with the system through a terminal environment.

The CLI should feel like an engineering assistant, not a programming tool.

The initial CLI scope contains:

- chat

---

# 7. Chat Command

Command:

```
piping-assistant chat
```

Purpose:

Start or continue an engineering session.


Behavior:

1. Load existing session state
2. Check unfinished tasks
3. Ask user whether to resume if applicable
4. Start AI interaction


Example:
```
> Calculate minimum wall thickness

Assistant:

I will evaluate ASME B31.3 thickness requirements.

Required information:

- design pressure
- temperature
- material
- diameter
```

---

## List Tasks




Output:


Shows:

- task ID

- completion state

Command:
```

```



Restores:

- conversation memory
- collected inputs
- graph traversal state
- execution progress


Example:
```
Task 001 restored.

Current state:

Waiting for:  
Design temperature
```

---

# 9. Report Commands


Generate report:
```
piping-assistant report generate <task_id> --format pdf
```



Supported formats:

- PDF
- HTML


Report workflow:

```
Execution Trace

↓

Report Data Model

↓

AI Presentation Layer

↓

Final Report
```



AI is allowed to:

- improve readability
- explain decisions
- create summaries


AI is not allowed to:

- change numbers
- modify formulas
- remove warnings
- alter traceability

# 16. Output Data Model

CLI responses should internally use structured objects.

Example:
```json
{
"status":"waiting_input",

"question":"Enter design pressure",

"required_by":"B313-304.1.1"
}
```

Benefits:

- future web UI
- API integration
- automated testing

---

# 17. Error Handling

The CLI displays system-level errors.

Example:
```
Node reference missing:  
  
B313-345
```
Engineering explanation is handled by AI.

The CLI should not attempt interpretation.

---

# 18. Storage Model

The CLI uses local storage.

It accesses:
```
/tasks

/reports

/nodes

/config
```

Storage responsibility remains with Storage Manager.

---

# 19. AI Connectivity

The CLI requires AI connectivity.

The deterministic engineering components remain independent.

Architecture:
```
AI:

navigation + explanation


Graph:

truth


Rules:

validity


Calculations:

results

```

---

# 20. Future Compatibility

The CLI should produce structured responses internally.

This allows future migration to:

- web interface
- desktop application
- API service

The CLI remains one presentation layer.

---

# 21. MVP Scope

Initial release:

Required:

- chat
- task management
- report generation

Optional:

- graph inspection
- node validation
- trace viewing

---

# 22. Design Principle

The CLI provides access to the engineering assistant.

It does not replace the engineering system.

Final architecture:
```
Graph defines truth.

Rules define validity.

Agents define navigation.

Calculations define results.

Reports explain decisions.

CLI provides interaction.
```

## Python implementation

> **Status:** MVP + optional inspection commands implemented in `cli/`.

| Command | Implementation |
|---------|----------------|
| `chat` | `cli/commands/chat.py` — interactive session via `ChatOrchestrator` |
| `task list` | `cli/commands/tasks.py` |
| `task resume <task_id>` | `cli/commands/tasks.py` |
| `report generate <task_id> --format pdf\|html\|md\|json` | `engine/reports/` via CLI (no placeholder) |
| `graph show <node_id>` | `cli/commands/graph.py` — tree from `standards/` dependencies |
| `node inspect <node_id>` | `cli/commands/nodes.py` |
| `node validate <node_id>` | `cli/commands/nodes.py` — schema + reference checks |
| `task trace <task_id>` | `cli/commands/tasks.py` — stored inputs/steps/outputs |
| `--debug-ai` | Global flag on `cli/app.py` |

| Module | Role |
|--------|------|
| `cli/orchestrator.py` | Wires Context, Intent, `engine.planner.Planner` (via PlannerAgent), Input agents + `TaskStateManager` |
| `cli/session_store.py` | Persists to `sessions/<session_id>/` |
| `cli/responses.py` | Structured `CLIResponse` objects (§16) |
| `cli/display.py` | Rich formatting |
| `config/config.yaml` | Default CLI settings (§15) |
| `config/loader.py` | `CLIConfig.load()` |
| `cli/standards_reader.py` | Read-only access to `standards/` node markdown |

Session storage layout:

```
sessions/<session_id>/
  conversation.json
  tasks.json
  reports/
```
