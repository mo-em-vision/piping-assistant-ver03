
# Execution Layer Design Documentation

## Purpose

This document defines the architecture, responsibilities, workflows, interfaces, and execution rules of the Deterministic Execution Layer.

The Execution Layer is the runtime engine responsible for executing engineering workflows after the Planner Layer and Graph Engine have determined what needs to be executed.

The Execution Layer converts:

- validated inputs
- dependency-resolved nodes
- formulas
- functions
- lookup operations

into deterministic engineering outputs.


Core principle:

> The Execution Layer executes engineering truth. It does not decide engineering truth.

---

# 1. Execution Layer Philosophy

The system follows a layered architecture:


```

User

↓

Planner Layer

↓

Graph Engine

↓

Execution Layer

↓

Rule Engine

↓

Report Generator

```


The Execution Layer is the deterministic runtime.

It ensures:


Same:

- node version
- input values
- execution configuration


Produces:


Same:

- outputs
- intermediate values
- warnings
- trace


---

# 2. Execution Layer Responsibilities


The Execution Layer manages:


## Workflow Execution

Responsible for:

- executing dependency-resolved workflows
- maintaining execution order
- passing outputs between nodes


---

## Calculation Execution

Responsible for:

- running approved calculation functions
- executing formulas
- storing intermediate values


---

## Lookup Execution

Responsible for:

- executing deterministic table lookups
- retrieving engineering values


---

## Trace Generation

Responsible for:

- recording every execution step
- preserving calculation history
- supporting report reconstruction


---

## Result Packaging

Responsible for creating standardized outputs from every node.


---

# 3. Execution Boundary


The Execution Layer does not:

- select engineering workflows
- interpret user intent
- decide applicable standards
- override engineering rules
- modify node content


The Execution Layer only executes approved workflows.


---

# 10. Calculation Precision

The Execution Layer uses:

Full precision internally.

Rules:

- no intermediate rounding
    
- no calculation rounding
    
- final formatting handled by reports
    

This prevents rounding errors propagating through dependent calculations.

# 12. Unit Management

Unit conversion is handled by the Unit Manager before execution.

Flow:

```
Original Input

↓

Unit Validation

↓

SI Conversion

↓

Calculation

```

The original value is preserved.

Example:

```
Input:

500 psi


Converted:

3447378 Pa
```

---

# 13. Input Validation

Validation occurs in stages:

```
Input Validation

↓

Unit Validation

↓

Calculation Validation
```

The Execution Layer verifies that required values exist before running.

---

# 14. Table Lookup Execution

Tables are handled by a dedicated Lookup Engine.

Example:

Material allowable stress:

```
Material

Temperature

↓

Table lookup

↓

```

# 15. Conditional Nodes

Conditional logic is evaluated through node-defined conditions.

Example:

```
IF temperature exceeds limit

perform additional check
```

The Execution Layer runs the condition.

The Rule Engine determines whether the condition is valid.

---

Flow:

```

↓

↓

↓
```

# 17. Handling Invalid Inputs

Temperature exceeds material limit.

Execution returns:

```yaml

```

---

# 18. User Corrections and Branching

If a user changes an input:

The previous execution is preserved.

Example:

Previous:

```
Temperature = 400 F
```

New:

```
Temperature = 600 F
```

System:

- stores previous execution
    
- creates new branch
    
- reruns affected dependencies
    

---

# 19. Result Invalidation

The Task State Manager owns invalidation.

Flow:

```
Input changed

↓

State Manager detects dependency

↓

Invalidate affected outputs

↓

Request Execution Layer rerun
```

Only dependent calculations are repeated.

## Python implementation

> **Status:** Implemented in `engine/executor/` and `engine/events/`. Graph Engine prerequisite in `engine/graph/`.

### Pipeline

```
ExecutionPlan (from GraphEngine)
        ↓
Executor.execute_plan()
        ↓
NodeRunner (per node: lookup | calculation)
        ↓
TaskStateManager (outputs, trace, status)
        ↓
EventLogger
```

### Modules

| Module | Role |
|--------|------|
| `engine/executor/executor.py` | `Executor`, `execute_workflow()` |
| `engine/executor/node_runner.py` | Single-node lifecycle (§7) |
| `engine/executor/calculation_engine.py` | Formula step execution |
| `engine/executor/lookup_engine.py` | Table lookup execution |
| `engine/executor/expression_evaluator.py` | Safe AST expression eval (§28) |
| `engine/executor/unit_manager.py` | SI conversion before execution (§12) |
| `engine/executor/functions.py` | Approved function registry |
| `engine/rules/rule_engine.py` | Minimal condition checks (not full Validation Layer) |
| `engine/events/event_logger.py` | Execution event audit trail |
| `models/execution.py` | `ExecutionPlan`, `NodeExecutionResult`, `ExecutionResult` |

### Scope notes

- **pipe_wall_thickness_design** — `B313-material-stress` lookup → `B313-304.1.1` calculation.
- Sequential execution only; parallel nodes deferred.
- Missing inputs pause task (`AWAITING_INPUT`); invalidation owned by State Manager.
- CLI wiring deferred to STEP 10 bootstrap.
