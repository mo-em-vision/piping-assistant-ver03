# System Architecture

## Purpose

This document defines the architecture of the engineering knowledge graph system.

The system converts engineering standards into executable dependency graphs where:

- standards paragraphs are represented as nodes
- dependencies form a directed graph
- calculations execute only after required information is collected
- reports are generated from the execution trace
- AI is responsible only for user interaction and explanation


---

# 1. Core Architecture Principle

The system is not a collection of workflows.

It is a dependency graph traversal engine.

A user request identifies an engineering intent.

The system discovers all required nodes, resolves dependencies, collects required information, executes calculations, and generates a traceable engineering report.


Flow:
```
User

в†“

AI Interaction Layer

в†“

Intent Identification

в†“

Root Node Discovery

в†“

Dependency Graph Traversal

в†“

Input Collection

в†“

DAG Execution

в†“

Report Generation

в†“

AI Explanation
```

# AI and Agent Responsibility Boundary  
  
  
The system separates engineering truth from intelligent navigation.  
  
  
The architecture uses three layers:  
  
  
## Layer 1 вЂ” Deterministic Graph Engine  
  
  
The graph engine is the source of engineering truth.  
  
  
It handles:  
  
  
- standard structure  
- paragraph nodes  
- formulas  
- tables  
- dependencies  
- execution order  
- versioning  
- traceability  
  
  
The graph engine must produce the same result when given the same inputs.  
  
  
Principle:  
  
  
"If the same input is provided, the same output must be produced."  
  
  
The graph engine does not reason.  
  
It executes defined engineering logic.  
  
  
  
---  
  
## Layer 2 вЂ” Rule Engine  
  
  
The rule engine performs deterministic validation.  
  
  
It handles:  
  
  
- unit validation  
- input constraints  
- temperature limits  
- material limitations  
- table range handling  
- assumptions  
- exceptions  
- warnings  
  
  
The rule engine verifies whether execution is allowed.  
  
  
It does not select calculations.  
  
  
  
---  
  
## Layer 3 вЂ” Lightweight Agents  
  
  
Agents operate only at the edges of the system.  
  
  
Agents provide navigation intelligence.  
  
  
They handle:  
  
  
- user intent understanding  
- node selection strategy  
- missing information resolution  
- multi-standard routing  
- explaining results  
  
  
Agents do not:  
  
  
- calculate values  
- select formulas  
- override rules  
- change outputs  
- modify standard interpretation  
  
  
Principle:  
  
  
Agents decide what to do next, but never what the truth is.

# 8. Graph Discovery

The system executes in two phases.

## Phase 1 вЂ” Discovery

Start:

User intent root node

Process:

1. Load root node
2. Read dependencies
3. Recursively expand dependencies
4. Build directed graph
5. Validate dependencies
6. Perform topological sorting

Output:

```
Execution graph
```

Example:
```
Integrity Check  
  
|  
  
+-- Wall Thickness  
  
| |  
  
| +-- Allowable Stress  
  
|  
  
+-- Pressure Test
```
---
# Phase 2 вЂ” Information Collection

Before calculation:

The system determines:

- required user inputs
- dependency outputs required
- unresolved conditions

The user is asked only for missing information.

# 10. Execution Engine

The calculation engine executes nodes.

Execution:

1. Load resolved inputs
2. Inject dependency outputs
3. Execute formula steps
4. Perform validations
5. Store outputs
6. Create trace event

Formula execution:
```yaml

formula:

 steps:

  - expression:

      "P*D/(2*(S*E*W+P*Y))"

    assign:

      thickness

```
Python executes the expression safely.

# 14. Report Generation  
  
  
Reports are generated from the engineering execution trace.  
  
The report is not created from final values only.  

Reports are generated from deterministic execution results.  
  
  
The report must preserve the reasoning path:  



```
standard paragraph в†’ dependency в†’ decision в†’ formula в†’ output в†’ conclusion  
```
  

  
  

The report pipeline:
  
  ```
Graph Execution Trace

в†“

Immutable Report Data Model

в†“

Report Template

в†“

Agent Presentation Layer

в†“

PDF / HTML Output
  ```

  The presentation agent may:  
  
  
- improve readability  
- organize explanations  
- summarize engineering reasoning  
  
  
The presentation agent may not:  
  
  
- modify values  
- modify formulas  
- remove warnings  
- change decisions  
- alter references  
  
  
The final report remains traceable to the original graph execution.

- execution order

- calculation steps

- decisions  
- references

---  
  
# AI Presentation Layer  
  
  
The AI receives the completed report data model.  
  
  
The AI is responsible for:  
  
  
- improving readability  
- explaining engineering reasoning  
- organizing sections  
- creating human-friendly summaries  
- improving wording  
  
  
The AI must not:  
  
  
- change inputs  
- change outputs  
- modify formulas  
- alter calculation results  
- remove warnings  
- change PASS/FAIL status  
- modify standard references  
  
  
The AI changes presentation only.

# Data Persistence Principles  
  
  
The system stores only information required for:  
  
  
- rerunning calculations  
- evaluating previous decisions  
- rebuilding reports  
- auditing execution  
  
  
Completed task:  
  
  
Stored:  
  
- report data model  
- graph version  
- inputs  
- outputs  
- decision trace  
  
  
Incomplete task:  
  
  
Stored:  
  
- user inputs  
- traversed nodes  
- current state  
  
  
Raw conversation history is stored separately.
