# System Architecture

## Purpose

This document is the **design hub** for the engineering knowledge graph system: repository layout, workflows/knowledge paths, one end-to-end lifecycle, and pointers to layer docs.

Policy and boundaries: [`docs/rules.md`](../rules.md). Component owns/must-not-own: [`3. component_responsibilities.md`](3.%20component_responsibilities.md).

---

# 1. Core principle

The system is a **dependency graph engine**, not a fixed workflow catalog.

Standards paragraphs, equations, lookups, and validation rules are separate authored nodes. Graph expansion determines the active subgraph; the Planner orders the next graph-derived ask; execution runs only after a resolved plan passes validation.

## End-to-end lifecycle

```
User → IntentAgent → Planner → Graph Engine (expand_workflow)
  → [input collection ↔ re-expand] until active subgraph resolved
  → ValidationEngine.validate_plan() → Executor.execute_plan()
  → ValidationEngine.validate_node() → ReportData → optional AI presentation
```

**Incremental expansion** — the Graph Engine does not build the complete execution graph before any input is collected. See [`5. workflow_design.md`](5.%20workflow_design.md) and [`14. graph_engine_design.md`](14.%20graph_engine_design.md).

**Live workflow UI** (parallel, not report pipeline): `engine/messaging/` + Flow Guidance (`GuidanceResolver`, `ResponseComposer`) — [`docs/rules.md`](../rules.md) §12, §21.

# 2. Layer summary

| Layer | Owns | Must not |
| --- | --- | --- |
| IntentAgent | User language → structured intent | Engineering requirements, prompt copy |
| Planner | Workflow selection, phase order, next graph-active ask | Active branches, formulas, compliance |
| Graph Engine | Expansion, reachability, branches, execution ordering | Formulas, lookups, validation, calculations |
| ValidationEngine | Compliance gate (plan + per-node) | Path selection, calculations |
| RuleEngine | Narrow runtime expression conditions only | Full compliance gate |
| Executor | Resolved `ExecutionPlan` | Raw-graph discovery, branch selection |
| State Manager | Persisted task state, invalidation, history | Engineering meaning |
| Messaging / Flow Guidance | Deterministic prompts, traversal narration | Which field is active |
| Report layer | ReportData, templates, optional AI polish | Engineering truth, live workflow prompts |

Full diagram: [`3. component_responsibilities.md`](3.%20component_responsibilities.md).

Agents decide what to do next, but never what the truth is.

# 6. Graph expansion and execution

1. Select workflow from intent or catalog.
2. `expand_workflow` with current Facts.
3. Expose unresolved gates or branch decisions only.
4. Collect permitted input; re-expand and replan.
5. Repeat until active subgraph and executable work are resolved.
6. `ValidationEngine.validate_plan()` then `Executor.execute_plan()`.

A **partially expanded graph** is not the same as a **resolved executable plan**.

Branch conditions during expansion use `engine/graph/conditions.py`. The Executor receives a resolved `ExecutionPlan` — see [`13. execution_layer_design.md`](13.%20execution_layer_design.md).

# 7. Report generation

```
Task state + execution/validation/graph traces + referenced knowledge
↓
ReportData builder (engine/reports/report_data.py)
↓
Deterministic report generator
↓
Optional AI presentation
↓
PDF / HTML
```

Flow Guidance is **not** in this pipeline. Detail: [`10. report_generation_design.md`](10.%20report_generation_design.md).

Report trace chain: `standard paragraph → dependency → decision → formula → output → conclusion`.

# 8. Legacy paths (compatibility only)

| Legacy | Current |
| --- | --- |
| `roots/.../root.md` | `workflows/{machine-key}.yaml` |
| Top-level `standards/` | `knowledge/standards/` |
| `type: definition` / `calculation` | Separate `paragraph`, `equation`, `lookup`, `validation_rule` |

`normalize_root_id()` in `engine/graph/graph_engine.py` maps historical `roots/` strings to workflow slugs for read-only compatibility. Do not author new `roots/` or `type: root` nodes.

# 9. Design goal

A traceable engineering reasoning engine. Every result can answer: why was this calculation performed, which paragraph drove the decision, and where did each value come from.
