# Report Generation Design Documentation

## Purpose

This document defines the architecture, workflow, data model, and responsibilities of the report generation system.

The report generation system converts deterministic engineering execution results into human-readable engineering documents while preserving:

- traceability
- auditability
- reproducibility
- engineering correctness


Core principle:

> The report explains engineering decisions, but it never changes engineering truth.


---

# 1. Report Generation Philosophy

The report system follows a data-first architecture.


Flow:
```
Calculation Execution

↓

Execution Trace

↓

Report Data Model

↓

Report Generator

↓

AI Presentation Layer

↓

Final Report
```

The engineering data is created before any language generation.


The AI layer is only responsible for:

- improving readability
- explaining decisions
- summarizing results
- creating understandable engineering narratives


The AI layer cannot:

- modify calculations
- change inputs
- remove warnings
- change formulas
- alter engineering conclusions


---

# 2. Report Architecture


Components:


## Execution Layer

Responsible for:

- running calculations
- collecting outputs
- recording intermediate values
- generating warnings


Output:

Report Data Object


---

## Report Data Layer

Stores the complete report-ready information.


Contains:

- task information
- graph traversal
- inputs
- formulas
- decisions
- calculations
- warnings
- limitations
- references


This layer allows reports to be regenerated without recalculating.


---

## Report Generator

Responsible for:

- applying templates
- formatting sections
- inserting tables
- inserting formulas
- adding figures
- generating PDF/HTML


---

## AI Presentation Layer

Receives:

Report Data


Produces:

Human-readable explanation.


The AI layer only changes presentation.

# 5. Standard Report Structure


All reports follow a fixed structure.


```
- Executive Summary
- Task Definition
- Applicable Standards
- Input Data
- Dependency Graph Traversal
- Engineering Decision Flow
- Calculation Details
- Results
- Limitations and Constraints
- Warnings
- User Overrides
- Conclusions
- References
- Appendix
```



---

# 6. Executive Summary


Contains:


- purpose
- requested task
- final status
- major warnings


Example:

```
Overall Result:

PASS WITH WARNING

Reason:

Flexibility analysis requires additional review.
```

---

# 7. Task Definition


Contains:


User request:
```
Verify pipe integrity
```



System interpretation:

```
Integrity assessment workflow selected
```
Selected root node:
```
integrity_check/root
```


---

# 8. Applicable Standards


Contains:

- standard references
- paragraphs used
- node versions


Example:

```
ASME B31.3

§304.1.1

§345
```
---  
  
# 9. Input Data Section  
  
  
Shows:  
  
  
User provided values:  
  
Example:
```
Design Pressure

Original:

500 psi

Calculation:

3447378 Pa
```


All units shown are the original user values.

Calculation units are included separately.


---

# 10. Dependency Graph Traversal


The report must show how the system decided what to execute.


Example:


```
Integrity Verification

↓
Wall Thickness Check

↓

Material Allowable Stress Lookup

↓

Pressure Test Verification
```



The traversal is generated from the actual graph execution.


---

# 11. Engineering Decision Flow


Conditional nodes must be explained.


Example:


```
Node:

B313-345

Reason:

Required because B313-345 depends on ordered wall thickness.

Previous output:

T_ordered
```

This explains why each node was selected.  
  
---  
  
# 12. Paragraph Traceability  
  
  
Each used paragraph includes:  
  
  
- paragraph ID  
- title  
- exact text  
- related notes  
  
  
Example:
```
Reference:

ASME B31.3 §304.1.1

Text:

[verbatim paragraph]

Source:

Node B313-304.1.1
```

  
---  
  
# 13. Formula Presentation  
  
  
Formulas are rendered using mathematical formatting.  
  
  
Example:  
  
  
\[  
t=\frac{PD}{2(SE+PY)}  
\]  
  
  
Below the formula:
```
Parameters:

P:

Design pressure

D:

Outside diameter

S:

Allowable stress
```

---  
  
# 14. Calculation Details  
  
  
Contains:  
  
  
- formula used  
- execution function  
- calculation steps  
- intermediate values  
  
  
Example:
```
Formula:

calculate_wall_thickness()

Step 1:

Validate inputs

Step 2:

Convert units

Step 3:

Calculate thickness

Step 4:

Check requirements
```

---  
  
# 15. Intermediate Values  
  
  
All intermediate values are included.  
  
  
No rounding is applied.  
  
  
Example:
```
Intermediate:

SE

PY

Denominator value

Calculated thickness
```

---  
  
# 16. Tables and Figures  
  
  
Reports may contain:  
  
  
- lookup tables  
- engineering figures  
- charts  
- calculation diagrams  
- dependency graphs  
  
  
---  
  
# 17. Results Section  
  
  
Contains:  
  
  
Final outputs:  
  
Example:
```
Required thickness:

0.065 in

Selected pipe:

Schedule 40

Status:

PASS
```

---  
  
# 18. Limitations and Constraints  
  
  
All node limitations are reported.  
  
  
Example:
```
Limitation:

Temperature exceeds preferred range.

Action:

Additional verification required.
```
---  
  
# 19. Warning Section  
  
  
Warnings are displayed separately.  
  
  
Example:
```
WARNING

User selected material outside recommended range.

Review required.
```

---  
  
# 20. User Overrides  
  
  
Overrides receive their own section.  
  
  
Example:
```
Override:

Temperature limitation

Original rule:

Reject

User decision:

Continue

Effect:

Previous calculations require review.
```

---  
  
# 21. Incomplete Task Reports  
  
  
Reports can be generated even if incomplete.  
  
  
Example:
```
Status:

Incomplete

Missing:

Design temperature

Completed nodes:

12/15
```
