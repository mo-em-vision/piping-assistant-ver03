
# Validation Layer Design Documentation

## Purpose

This document defines the architecture, responsibilities, workflows, interfaces, and validation rules of the Validation Layer.

The Validation Layer is the deterministic engineering compliance layer of the system.

Its responsibility:

> Determine whether a planned engineering operation is allowed.

It does not:

- select workflows
- calculate formulas
- generate execution paths
- replace engineering judgement

---

# 1. Validation Layer Philosophy

The system follows:

```

Planner Layer

↓

Graph Engine

↓

Validation Layer

↓

Execution Layer

↓

Report Generator

```


The Validation Layer answers:

"Can this operation proceed?"


The Execution Layer answers:

"How is it calculated?"

---

# 2. Core Principle


> Validation enforces engineering rules.


The Validation Layer protects deterministic engineering execution by ensuring:

- inputs are valid
- units are consistent
- limits are respected
- dependencies are satisfied
- exceptions are recorded

---

# 3. Validation Responsibilities


The Validation Layer manages:


## Input Validation

Checks:

- required inputs
- data type
- valid ranges
- missing information


---

## Unit Validation

Checks:

- unit compatibility
- dimensional correctness
- conversion requirements


---

## Engineering Validation

Checks:

- material limits
- temperature restrictions
- pressure limits
- code constraints


---

## Dependency Validation

Checks:

- required outputs exist
- previous calculations are valid


---

## Conflict Detection

Checks:

- contradictory inputs
- changed parameters
- affected calculations


---

```

```


Each validator has a specific responsibility.

# 7. Input Validation


Input validation checks:


Example:


User enters:


```

Pressure = abc

````


Result:


```yaml
status:

error


message:

Unknown value


action:

Request valid input

suggestions:

- 500 psi
- 600 psi

````

The system should assist the user when possible.

---

# 8. Unit Validation

The Unit Validator checks:

- unit conversion
    
- dimensional correctness
    
- preferred units
    

Example:

```
500 psi
```

Converted:

```
3447378 Pa
```

The original value is preserved.

---

# 9. Unit Data Storage

Validation records:

```yaml
parameter:

pressure


original:

500 psi


converted:

3447378 Pa


source:

user_input

```

This supports:

- audit
    
- traceability
    
- report generation
    

---

# 10. Engineering Validation

Engineering validation checks:

Examples:

- material temperature limits
    
- allowable stress range
    
- pressure limitations
    
- design restrictions
    

Rules may come from:

- node limitations
    
- tables
    
- standard references
    

---

# 11. Rule Storage Model

Rules may exist in:

## Nodes

Example:

```yaml
limitations:

maximum_temperature:

500F

```

---

## Separate Rule Files

Example:

```
rules/

materials.yaml

temperature.yaml

```

The system supports both.

---

# 12. Rule Priority

Rules return:

- errors
    
- warnings
    
- information
    

Example:

```yaml
severity:

warning
```

Multiple rules may be returned.

The system does not hide conflicting rules.

---

# 13. Hard Failures

Example:

Material cannot operate at required temperature.

Validation returns:

```yaml
status:

error


reason:

temperature exceeds limit
```

The workflow pauses.

---

# 14. User Overrides

Overrides are allowed with confirmation.

Example:

Rule:

```
temperature exceeds recommended limit
```

User:

```
continue
```

System:

- records override
    
- continues execution
    
- marks warning
    

Report:

```
WARNING:
Engineering limitation overridden by user.
```

---

# 15. Default Values

Nodes may define defaults.

Example:

```yaml
default:

E:

1.0

```

Default handling:

The system presents the default to the user.

Example:

```
Elastic modulus not provided.

Default value:
1.0

Confirm?
```

After confirmation:

- value applied
    
- source recorded
    

---

# 16. Uncertainty and Ranges

Ranges are supported.

Example:

```
170-180 MPa
```

Validation:

- informs user
    
- explains limitation
    
- requests confirmation
    

The system does not silently select values.

---

# 17. Table Validation

Table lookup rules are handled by the Lookup Engine.

Example:

Requested temperature:

```
375 C
```

Available:

```
300 C
400 C
```

Selection:

lower available value:

```
300 C
```

No interpolation is performed.

---

# 18. Dependency Validation

Dependency validation checks:

Example:

Node:

```
Pressure Test
```

requires:

```
ordered thickness
```

Validation:

```
Is ordered thickness available?
```

---

# 19. Contradictory Inputs

Example:

Previous:

```
Material = A106B
```

New:

```
Material = Stainless Steel
```

Validation detects conflict.

Process:

```
Detect conflict

↓

Inform user

↓

Prepare new calculation branch

↓

Show changed parameters

↓

Request confirmation
```

---

# 20. Calculation Branch Changes

When parameters change:

The system creates a new calculation branch.

Previous results remain available.

Example:

```
Calculation A

temperature = 400F


Calculation B

temperature = 600F

```

Both are preserved.

---

# 21. Previous Calculation Validity

The Validation Layer identifies impacted conditions.

Example:

```
Previous calculation may no longer be valid

Reason:

input temperature changed
```

The State Manager controls rerun decisions.

The Validation Layer does not require full version storage.

```yaml

```

---

# 27. Multiple Standards

When multiple standards conflict:

The system:

- detects conflict
    
- reports conflict
    
- asks user decision
    

Example:

```
ASME rule differs from API rule.

Select applicable standard.
```

---

# 28. Report Integration

- warnings
    
- overrides

- special limitations
    

Example:

```

User override accepted.

```

# 31. Final Architecture Principle

The Validation Layer is the compliance gate of the system.

It ensures:

```
Valid inputs

+

Valid dependencies

+

Valid engineering conditions

=

Safe deterministic execution
```

Final principle:

> The Validation Layer determines whether a planned engineering operation is allowed, but it never performs the calculation itself.
