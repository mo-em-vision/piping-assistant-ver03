
# Graph Engine Design Documentation

## Purpose

This document defines the architecture, responsibilities, workflows, and interfaces of the Graph Engine.

The Graph Engine is the deterministic dependency resolution layer of the system.

It is responsible for answering:

> What needs to happen?

It does not answer:

> Is it allowed?

That responsibility belongs to the **Validation Layer** (`ValidationEngine`).

`RuleEngine` handles narrow node-level condition evaluation only.

It does not answer:

> How is it calculated?

That responsibility belongs to the Execution Layer.


Core principle:

> The graph decides what must happen, but not whether it is allowed or how it is calculated.

The Graph Engine must **not** store workflow-specific traversal narration or user-facing guidance copy. Intrinsic node assets (paragraph text, equation metadata, `ai_hints`) remain on knowledge nodes. Flow narration is presentation data resolved by `GuidanceResolver` (`docs/rules.md` §21).

# 9. Node Discovery Strategy

The Graph Engine uses a hybrid discovery approach with **two distinct modes**. Do not mix them in one call.

## Specific lookup (`workflow` slug provided)

When the caller passes an explicit workflow slug or intent id (`discover_roots(..., workflow="pipe_wall_thickness_design")`):

- Treat the call as a **lookup**, not a scan.
- Return only workflow roots whose `slug`, `engineering_intent`, or workflow `node_id` matches the reference (including partial slug containment for typo tolerance).
- Return an **empty list** when nothing matches — never fall back to listing every workflow at a low baseline confidence.
- **Ignore `keywords`** in this mode so a bad slug cannot be widened into broad discovery.

## Broad discovery (`workflow` omitted)

When no explicit slug is provided (`discover_roots(reader)` or `workflow=None`):

- Scan all compiled workflow nodes.
- Assign baseline discovery confidence (`0.4`) to each candidate.
- Boost confidence (`0.75`) when `keywords` match workflow metadata (`engineering_intent`, title, slug, purpose).
- Legacy stub roots (not yet implemented) are included only in this mode when keywords match.

The Planner uses broad discovery when `intent.workflow` and `intent.intent` are both unset, and specific lookup when either carries a slug.

It uses:

## Section-Level Metadata

For broad discovery.

Example:

```
pipe integrity
pressure testing
wall thickness
```

---

## Node Relationships

For precise traversal.

The system does not require keywords for every paragraph.

Relationships remain the primary navigation method.

# 32. Expansion Gating and Parameter Registry

Before user parameters are requested, the graph evaluates **expansion assumptions** declared on definition nodes (e.g. `B313-304.1.1`).

## `required_for_expansion`

Assumption and interaction frontmatter may set `required_for_expansion: true`. The assumption checker (`engine/graph/assumption_checker.py`) evaluates these fields before path expansion.

For pipe wall thickness:

- `straight_pipe_section` — must be confirmed `true` before the wall-thickness path expands
- `pressure_loading` — must be `internal_pressure` or `external_pressure` to select `304.1.2-a` vs `B313-304.1.3`

`GraphEngine.expansion_gate_ready()` returns `False` until both are satisfied.

While the gate is closed:

- `required_user_inputs()` returns `[]`
- parameter extraction in the CLI is limited to expansion/path fields (`allowed_fields_for_phase`)

## Parameter registry

`GraphEngine.seed_parameter_registry()` builds `ParameterDescriptor` entries for `Task.parameter_registry` **after** the expansion gate passes. Descriptors carry provenance metadata (`input_id`, symbol, description, `introduced_at_node`, unit, optional resolution fields). They inform questioning and traceability; they are **not** stored as pending task input values.

### Micro-graph path (canonical)

When the workflow resolves through the compiled micro-graph (`GraphEngine.uses_micro_graph()` is true — normal operation with graph cache):

1. `GraphEngine.seed_parameter_registry()` delegates to `MicroGraphEngine.seed_parameter_registry()`.
2. The micro engine re-checks `expansion_gate_ready()`; if false, returns `{}`.
3. It runs `expand_workflow()` and walks **`active_nodes`**.
4. For each active compiled **`PARAM-*`** node (skipping UI gate parameters: assumptions and interactions via `is_ui_parameter()`):
   - Registry key = parameter `key` (or `input_id` when authored).
   - Descriptor fields are copied from parameter node metadata: `symbol` (or `canonical_symbol`), `title`/`name`/`description`, `unit` (or `canonical_unit` via dimension inheritance), `defined_in` / `introduced_by`, `concept_id`.
   - `prepare_parameter_metadata()` normalizes `canonical_symbol` → `symbol`, `name` → `title`, `introduced_by` → `defined_in`, lifts nested `metadata:` blocks, and inherits `canonical_unit` / `allowed_units` from referenced `DIM-*` nodes when absent.
5. Lookup-derived and equation-output parameters on the active path may appear in the registry for provenance even when `required_user_inputs()` does not ask users for them directly.

While the gate is closed, `seed_parameter_registry()` returns `{}` (same gating contract as `required_user_inputs()`).

### Legacy fallback (deprecated)

`engine/graph/parameter_registry.py` → `seed_parameter_registry(reader, execution_order=…)` remains only when:

- `VER03_LEGACY_GRAPH_TRAVERSAL=1`, or
- the compiled micro-graph cache is unavailable,

and `GraphEngine` falls back to legacy `depends_on` traversal. That path reads **definition-node nomenclature** (e.g. `B313-304.1.1` `nomenclature[]` with `introduced_here`) plus calculation-node `inputs[]` on the active branch. It emits a `DeprecationWarning` and is **not** the canonical source of truth for micro-graph workflows.

## Conditional expansion

`depends_on.when` clauses and node `interactions` prune execution paths (e.g. internal vs external pressure). Skipped nodes whose `when` clause is unmet are promoted to `missing_assumptions` until the controlling field is confirmed.

## Provisional execution assumptions

At expansion, the graph may seed provisional values (e.g. `thin_wall: true` on `304.1.2-a`) that are re-evaluated after calculation.

## Python implementation

> **Status:** Minimal MVP in `engine/graph/graph_engine.py`. Produces `ExecutionPlan` for the Execution Layer.

### API

```python
from engine.graph import GraphEngine, normalize_root_id

candidates = GraphEngine().discover_roots(reader, workflow="pipe_wall_thickness_design")
required = GraphEngine().required_user_inputs("pipe_wall_thickness_design", reader)
slug = normalize_root_id("roots/pipe_wall_thickness_design/root.md")

plan = GraphEngine().build_plan(
    task_id="...",
    root_id="pipe_wall_thickness_design",
    inputs=task.inputs,
    reader=standards_reader,
)
```

### Behavior (MVP)

1. **Discovery** — two modes: **specific lookup** when `workflow` is set (matching roots only; empty on unknown slug; keywords ignored), and **broad discovery** when `workflow` is omitted (scan all workflow nodes; keyword boosts). Return `WorkflowCandidate` list.
2. **Expansion gating** — `expansion_gate_ready()` blocks `required_user_inputs()` until `straight_pipe_section` and `pressure_loading` are confirmed.
3. **Parameter registry** — after the gate passes, `seed_parameter_registry()` seeds descriptors from **active compiled `PARAM-*` nodes** on the expanded path (`MicroGraphEngine`). Legacy definition-node nomenclature seeding (`engine/graph/parameter_registry.py`) applies only when legacy traversal is enabled or the graph cache is absent.
4. **Required inputs** — walk `build_plan` node set; collect required `user_input` fields (used by Planner); empty while expansion gate is closed.
5. **Plan build** — load root and recursively collect `depends_on` nodes via `StandardsReader`; honor `when` clauses for conditional branches.
6. Build `GraphEdge` list with dependency types.
7. Topological sort → `execution_order` (dependencies first).
8. Attach `GraphVersion` with node list and versions.
9. Return `ExecutionPlan` — no formula execution, no rule validation.

| Module | Role |

### Scope notes

- **Conditional path pruning** implemented for `pressure_loading` (internal → `304.1.2-a`, external → `B313-304.1.3`) and expansion assumptions on `B313-304.1.1`.
- **Graph-driven paths (§13)** — active branches and parameter asks come from expansion + `required_user_inputs()`, not hardcoded workflow field lists in engine/API/UI. Navigation `runtime.yaml` defines phase order and gate fields only. Replan after each confirmed input. See `docs/rules.md` §13 and `docs/desktopApp/05_backend_ui_contract.md`.
- Dynamic discovery / multi-workflow merging deferred (stub candidates for integrity/pressure test keywords).
- Cycle detection raises `GraphCycleError`.
