
# Graph Engine Design Documentation

## Purpose

This document defines the architecture, responsibilities, workflows, and interfaces of the Graph Engine.

The Graph Engine is the deterministic dependency resolution layer of the system.

It is responsible for answering:

> What needs to happen?

It does not answer:

> Is it allowed?

That responsibility belongs to the Rule Engine.

It does not answer:

> How is it calculated?

That responsibility belongs to the Execution Layer.


Core principle:

> The graph decides what must happen, but not whether it is allowed or how it is calculated.

---

# 1. Graph Engine Philosophy

The Graph Engine represents engineering dependency truth.


Architecture:

```

User

↓

Planner Layer

↓

Graph Engine

↓

Rule Engine

↓

Execution Layer

↓

Report Generator

```


The Graph Engine converts a user intent into a deterministic execution structure.

# 3. Graph Boundary


The Graph Engine does:


- dependency resolution
- path generation
- traversal
- ordering
- graph version management


The Graph Engine does not:


- execute formulas
- validate engineering limits
- calculate results
- replace standards

## Python implementation

> **Status:** Minimal MVP in `engine/graph/graph_engine.py`. Produces `ExecutionPlan` for the Execution Layer.

### API

```python
from engine.graph import GraphEngine, normalize_root_id

candidates = GraphEngine().discover_roots(reader, workflow="pipe_wall_thickness_design")
required = GraphEngine().required_user_inputs("pipe_wall_thickness_design", reader)
slug = normalize_root_id("roots/pipe_wall_thickness_design/root.md")

plan = GraphEngine().build_plan(
    task_id="...",
    root_id="pipe_wall_thickness_design",
    inputs=task.inputs,
    reader=standards_reader,
)
```

### Behavior (MVP)

1. **Discovery** — scan `standards/.../roots/`; match `engineering_intent`, title, keywords; return `WorkflowCandidate` list.
2. **Required inputs** — walk `build_plan` node set; collect required `user_input` fields (used by Planner).
3. **Plan build** — load root and recursively collect `depends_on` nodes via `StandardsReader`.
4. Build `GraphEdge` list with dependency types.
5. Topological sort → `execution_order` (dependencies first).
6. Attach `GraphVersion` with node list and versions.
7. Return `ExecutionPlan` — no formula execution, no rule validation.

### Scope notes

- Conditional path pruning not implemented — all declared dependencies included.
- Dynamic discovery / multi-workflow merging deferred (stub candidates for integrity/pressure test keywords).
- Cycle detection raises `GraphCycleError`.
