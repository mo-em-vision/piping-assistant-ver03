# AI Agent Design Documentation

## Purpose

This document defines the responsibility, behavior, and boundaries of AI agents.

Agents provide navigation and communication intelligence.

They do not define engineering truth.

Core principle:

> Agents decide what to do next, but never what the truth is.

---

# 1. Agent Architecture

The system uses specialized lightweight agents.

Structure:

```
ai/

agents/

- intent_agent.py
- planner_agent.py
- input_agent.py
- routing_agent.py
- synthesis_agent.py
- context_agent.py
```


Each agent has one responsibility.

---

## Python implementation

> **Status:** Implemented in `ai/agents/`. OpenAI via `ai/client.py`. Structured outputs in `models/agent.py`.

| Agent | Module | Deterministic behavior | LLM fallback |
|-------|--------|------------------------|--------------|
| Intent | `intent_agent.py` | Wraps `engine.router`; enriches `pipe_wall_thickness_design` | Unmatched requests |
| Planner | `planner_agent.py` → `engine/planner/planner.py` | `pipe_wall_thickness_design` navigation + missing inputs | Ambiguous / unknown intents (LLM) |
| Input | `input_agent.py` | Current-phase missing fields from `NavigationPlan.phase_missing`; no LLM during deterministic phases | Wording enrichment only when phase has no structured prompt |
| Routing | `routing_agent.py` | ASME B31.3 vs API 570 for inspection | Ambiguous standards |
| Context | `context_agent.py` | Off-topic keyword detection; **no LLM** while an active task is in progress | Unrelated input with no active task |
| Synthesis | `synthesis_agent.py` | Fallback markdown from `ReportData` | Presentation prose |

Canonical intent/workflow: `pipe_wall_thickness_design` (matches router and standards root).

Environment: `OPENAI_API_KEY`, optional `OPENAI_MODEL` (default `gpt-4o-mini`). See `.env.example`.

Agents may access State Manager and future Graph/Reference/Report engines via caller wiring — they do not execute formulas or modify engineering values.

# 2. Intent Agent

## Purpose

Understand user requests and classify engineering intent.

Responsibilities:

- identify user objective
- identify engineering domain
- identify possible standards
- identify missing context
- propose initial root nodes


Example:

User:

"Design pipe thickness for refinery service"


Output:

intent:

pipe_wall_thickness_design

domain:

piping

possible_standards:

ASME B31.3

root_nodes:

roots/pipe_wall_thickness_design/root.md


---

# 3. Planner Agent

## Purpose

Convert intent into graph navigation strategy.

> **Implementation:** Deterministic planning lives in `engine/planner/Planner`. `PlannerAgent` delegates to it and uses LLM fallback only when confidence is low or roots are ambiguous.


Responsibilities:

- select candidate root nodes (via `GraphEngine.discover_roots`)
- rank possible workflows
- propose execution paths
- derive missing inputs from graph metadata (no calculation)


Example:

Output:

priority:

1. material stress evaluation

2. pressure design / wall thickness

3. thin-wall applicability check


The planner does not calculate.

---

# 4. Missing Information Agent

## Purpose

Collect required inputs **in planner phase order**. User-facing questions are built **deterministically** from graph metadata — not by the LLM — whenever the active phase has a finite choice set or a formula-parameter template.


Responsibilities:

- identify missing values for the **current navigation phase only** (`NavigationPlan.phase_missing[current_phase]`)
- surface numbered options when node `interactions` or assumptions define `options` / yes-no gates
- never call the LLM to rephrase expansion, path, or coefficient-confirmation prompts
- never extract or store fields outside the active phase (orchestrator uses `allowed_fields_for_phase`)


## LLM usage boundary (chat)

| When | LLM used? |
|------|-----------|
| First message — classify intent / find workflow root | Yes, if `engine.router` cannot match keywords |
| Active task — confirm assumption, path decision, provide P/D/material | **No** |
| Active task — unrelated or mixed off-topic message | Yes (context switch detection) |
| Ambiguous workflow / multiple roots | Yes (planner or intent clarify) |

Deterministic prompt builders (no LLM):

- `engine/messaging/step_prompt.py` — expansion assumptions, path decisions, joint category, default confirmations
- `engine/messaging/formula_parameter_prompt.py` — calculation-node formula + known/missing parameters
- `engine/messaging/prompt_format.py` — numbered choices and per-parameter line layout
- `ai/input_extractor.py` — parse user replies (`1`, `option 2`, `yes`, `P: 8 bar`, etc.)


## Phased examples

**Expansion (first turn):**

```
input_id: straight_pipe_section

Choose one:
  1. Yes — straight pipe section (supported)
  2. No — fitting or bend (not yet supported)
```

**Path decision:**

```
input_id: pressure_loading

Choose one:
  1. Internal pressure — design per §304.1.2 (node 304.1.2-a)
  2. External pressure — design per §304.1.3 (node B313-304.1.3)
```

**Parameter gathering (after expansion gate):**

```
Formula:
  t = PD / (2(SEW + PY))

Known parameters:
  P
    Description: Internal design gage pressure
    Value: 8 bar

Missing parameters:
  D
    Description: Outside diameter of pipe
    Needed: requires NPS for lookup …
    Choose one:
      1. Nominal pipe size (NPS) — look up outside diameter per ASME B36.10
      2. Outside diameter — enter D directly (mm or in)
```

The Input Agent exposes only the **current phase** missing list to the orchestrator. It does not merge all future fields into one flat request on the first turn.

# 5. Context Agent

## Purpose

Maintain conversation continuity during multi-turn input collection.


While a task is **active**, the Context Agent uses keyword heuristics only. It does **not** call the LLM on every follow-up message — step-by-step replies (`yes`, `1`, `internal pressure`, `P: 500 psi`) stay on the deterministic path.


If unrelated user messages occur:

Example:

Active task:

pipe integrity calculation


User:

"What is today's weather?"


System:

- preserve current task
- detect context switch
- ask whether to continue or switch


Tasks are never silently abandoned.

# 6. Multi Standard Routing Agent

When multiple standards apply:

Example:

Request:

"Inspect this pipe"


Possible:

- ASME B31.3
- API 570


Agent presents options.

User selects design basis.

---

# 7. Agent Tool Access

Agents may access:

- Graph Engine
- State Manager
- Reference Resolver
- Report Generator


Agents may not:

- execute formulas directly
- modify calculations
- bypass rules
- change engineering values


---

# 8. Agent Memory

Agents can access:

- current session memory
- previous completed tasks
- historical reports
- relevant execution data


Memory is controlled by the State Manager.

---

# 9. Agent Output Format

Agents internally produce structured outputs.

Example:

action:

request_input


input:

temperature


reason:

required by node B313-304.1.1


Human-readable responses are generated afterwards.

---

# 10. Confidence Handling

If confidence is low:

Agent must:

- ask clarification
- provide options


Agents must not guess engineering intent.

---

# 11. Agent Error Handling

If agent selects an incorrect path:

System validates.

User may request alternative search.

Alternative node selection is recorded.

Example:

Selected node:

API 570 inspection

Alternative chosen:

ASME B31.3 integrity check


The final report records the selected path.

---

# 12. Report Synthesis Agent

Purpose:

Convert report data into readable output.


Allowed:

- improve wording
- explain decisions
- create summaries


Forbidden:

- changing values
- changing formulas
- removing warnings
- altering traceability


---

# 13. User Overrides

If a rule is overridden:

Agent first confirms with user.

Example:

Temperature exceeds material limit.

User override:

accepted.


The system records:

- violated rule
- user decision
- reason


Override is always shown in the final report.

# Final Principle

Graph defines truth.

Rules define validity.

Agents define navigation.

Calculations define results.

Reports explain what happened.
