
# Planner Layer Design Documentation

## Purpose

This document defines the architecture, responsibilities, workflows, and boundaries of the Planner Layer.

The Planner Layer is a lightweight agent layer responsible for deciding:

- what the user is asking
- where to start searching
- what workflows may be required
- what information is missing
- what questions should be asked next
- what alternative paths exist


The Planner Layer does not determine engineering truth.


Core principle:

> Agents decide what to do next, but never what the truth is.


Engineering truth remains controlled by:

- Deterministic Graph Engine
- Rule Engine
- Calculation Engine


---

# 1. Planner Layer Philosophy

The Planner represents the reasoning/navigation layer of the system.


Architecture:

```

User

↓

Planner Agent

↓

Graph Engine

↓

Rule Engine

↓

Calculation Engine

↓

Report Generator

```


The Planner behaves like a senior engineer deciding where to look in the engineering knowledge graph.

However:

The Planner does not replace engineering standards.

The Planner does not perform calculations.

The Planner does not modify rules.

# 3. Planner Boundary


The Planner must never:


- execute formulas
- perform engineering calculations
- change node definitions
- modify standards
- override rules
- determine compliance directly
- **own user-facing parameter prompt text** (wording belongs in `engine/messaging/`; see `docs/rules.md` §12)


The Planner only creates navigation decisions.

# 11. Missing Information Strategy

The Planner identifies minimum information required **in phased order**.

The Planner does not randomly request every input or merge all missing fields into one flat list.

It analyzes dependencies and the current navigation phase.

## Phased questioning

`build_phased_navigation()` (in `engine/graph/navigation_phases.py`) partitions missing fields into:

1. **expansion_assumptions** — `straight_pipe_section` (blocks expansion when false)
2. **path_decisions** — `pressure_loading` (internal vs external branch)
3. **parameter_gathering** — P, NPS/D, material, temperature
4. **coefficient_resolution** — joint category, E, W, Y defaults/lookups
5. **execution_assumptions** — corrosion allowance (c)

`NavigationPlan` carries:

- `current_phase` — active phase enum
- `phase_missing` — per-phase field lists
- `missing_assumptions` — expansion/path fields not yet confirmed
- `missing_inputs` — user-parameter fields (empty until expansion gate passes)
- `questions` — questions for the **current phase only**

## Pipe wall thickness example

**First turn** (no inputs):

```
current_phase: expansion_assumptions
phase_missing.expansion_assumptions: [straight_pipe_section]
missing_inputs: []
```

User sees numbered yes/no for straight pipe (step prompt). `pressure_loading` is planned for the next phase but is **not** requested until `straight_pipe_section` is confirmed.

**After straight section confirmed**:

```
current_phase: path_decisions
phase_missing.path_decisions: [pressure_loading]
```

User sees numbered internal vs external pressure choices (step prompt).

**After pressure_loading confirmed**:

```
current_phase: parameter_gathering
phase_missing.parameter_gathering: [design_pressure, nominal_pipe_size, material, design_temperature]
```

User sees formula parameter prompt for §304.1.2 with known/missing blocks per symbol.

The goal:

Collect only the information needed to unlock the **next** graph step.

# 19. Planner Prompt Structure

**Two prompt layers — do not mix them.**

| Layer | Location | Purpose |
| --- | --- | --- |
| **Deterministic workflow asks** | `engine/messaging/` | Desktop/CLI parameter questions, step prompts, formula blocks, default catalog (`workflow_parameter_prompts.py`). Resolved via `parameter_input_prompt.build_parameter_input_prompt()`. |
| **LLM planner agent** | `ai/prompts/planner/` | Intent detection, ambiguity, node-selection stubs for the Planner **agent** only. |

The Python `Planner` class (`engine/planner/planner.py`) must not store parameter prompt strings. It passes field ids and phases; messaging resolves copy at ask time.

LLM stubs (agent only):

```
ai/prompts/planner/
    intent_detection.md
    node_selection.md
    question_generation.md
    ambiguity_resolution.md
```

# 24. Final Architecture Principle

The Planner acts as the navigation intelligence layer.

System model:

```
Planner:

Decides where to look


Graph:

Defines dependencies


Rules:

Define allowed operations


Calculations:

Produce results


Reports:

Explain results
```

Final rule:

> The Planner may suggest the path, but the graph and rules determine the engineering outcome.

## Python implementation

> **Status:** Implemented in `engine/planner/`. `PlannerAgent` in `ai/agents/` is a thin adapter with LLM fallback.

### Pipeline

```
IntentResult (from IntentAgent)
        ↓
Planner.plan()
        ↓
GraphEngine.discover_roots() + build_plan preview
        ↓
NavigationPlan (missing inputs, selected root, questions)
        ↓
InputAgent (current-phase missing list)
        ↓
engine/messaging/step_prompt | formula_parameter_prompt (numbered options, no LLM)
```

### Modules

| Module | Role |
|--------|------|
| `engine/planner/planner.py` | `Planner` — navigation coordinator |
| `engine/planner/tools.py` | `GraphTools`, `StateTools`, `RuleTools` |
| `engine/planner/engineering_plan_builder.py` | `EngineeringPlan` assembly for pipe wall (and extensible workflows) |
| `engine/planner/planner_traversal.py` | `PlannerTraversalState` — debug snapshot of planner graph walk |
| `engine/planner/plan_inspector.py` | `planner_inspector_summary`, `engineering_plan_view`, `planner_inspector_summary_for_task` |
| `engine/planner/plan_validation.py` | `validate_engineering_plan`, `validate_engineering_plan_dict` |
| `engine/planner/plan_dependencies.py` | Central `engineering_plan.dependencies` from requirement `depends_on` |
| `engine/planner/plan_phases.py` | `derive_plan_phases`, `derive_input_strategy` |
| `engine/planner/activation_conditions.py` | Conditional requirement activation (`active` / `conditional` / `not_applicable`) |
| `models/engineering_plan.py` | `EngineeringPlan`, `PlannerTraversalState`, requirements, phases |
| `models/planning.py` | `NavigationPlan`, `NavigationPhase`, `WorkflowCandidate`, `StructuredIntent` |
| `engine/graph/navigation_phases.py` | `build_phased_navigation()`, `allowed_fields_for_phase()` |
| `ai/agents/planner_agent.py` | Delegates to `Planner`; LLM fallback on ambiguity |
| `engine/messaging/step_prompt.py` | Deterministic assumption/path/confirmation prompts |
| `engine/messaging/parameter_input_prompt.py` | Resolves parameter ask copy (interaction → graph → equation → defaults) |
| `engine/messaging/workflow_parameter_prompts.py` | Default catalog for workflow parameters (e.g. material search hint) |
| `engine/messaging/formula_parameter_prompt.py` | Formula + per-symbol known/missing blocks |
| `ai/prompts/planner/*.md` | Focused prompt stubs (§19) |

### Scope notes

- **pipe_wall_thickness_design** fully supported with phased navigation. The CLI asks **one phase at a time**: first `straight_pipe_section` (numbered yes/no), then `pressure_loading` (numbered internal/external), then formula parameters for §304.1.2 or §304.1.3. Parameter fields (`material`, `design_temperature`, etc.) appear only after the expansion gate passes.
- Ambiguous requests return `CLARIFY` with alternative workflow stubs.
- Planner decisions logged as `EventType.PLANNER_DECISION`; stored in `task.outputs.planning_summary`.
- No formula execution; graph execution deferred to `Executor` (STEP 4).
