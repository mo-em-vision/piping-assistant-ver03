
# Planner Layer Design Documentation

## Purpose

This document defines the architecture, responsibilities, workflows, and boundaries of the Planner Layer.

The Planner Layer is a lightweight agent layer responsible for deciding:

- what the user is asking
- where to start searching
- what workflows may be required
- what information is missing
- what questions should be asked next
- what alternative paths exist


The Planner Layer does not determine engineering truth.


Core principle:

> Agents decide what to do next, but never what the truth is.


Engineering truth remains controlled by:

- Deterministic Graph Engine
- Rule Engine
- Calculation Engine


---

# 1. Planner Layer Philosophy

The Planner represents the reasoning/navigation layer of the system.


Architecture:

```

User

↓

Planner Agent

↓

Graph Engine

↓

Rule Engine

↓

Calculation Engine

↓

Report Generator

```


The Planner behaves like a senior engineer deciding where to look in the engineering knowledge graph.

However:

The Planner does not replace engineering standards.

The Planner does not perform calculations.

The Planner does not modify rules.


---

The Planner is responsible for:

Example:


User:

```

```


Planner:

```

object:  
pipe

action:  
verification

domain:  
piping integrity

confidence:  
0.85
```

---

## Workflow Discovery

Determine possible engineering workflows.


Example:


User:

```

Verify pipe integrity

```


Planner:

Possible workflows:

```

- wall thickness verification
    
- pressure test verification
    
- flexibility assessment
    
- inspection requirements
    

```


---

## Node Routing

Identify relevant graph entry points.


Example:


```

integrity_check/root

```


The Planner selects where the Graph Engine should begin traversal.


---

## Missing Information Resolution

Determine what information is required to continue.


Example:


```

Required:

design pressure

temperature

material

diameter

```


---

## Multi-standard Routing

Identify relevant standards.


Example:


User:

```

Check vessel

```


Planner:

Possible standards:

```

ASME B31.3

API 570

BPVC Section VIII

```


The Planner may request user confirmation.


---

# 3. Planner Boundary


The Planner must never:


- execute formulas
- perform engineering calculations
- change node definitions
- modify standards
- override rules
- determine compliance directly


The Planner only creates navigation decisions.


---

# 6. Planner and Graph Engine Interaction

The Planner does not directly execute graph traversal.

Flow:

```
Planner

↓

Candidate node request

↓

Graph Engine search

↓

Available nodes

↓

Planner selects workflow

↓

Graph Engine executes traversal
```

The Graph Engine remains responsible for:

- dependencies
    
- ordering
    
- execution graph
    

---

# 7. Graph Search Strategy

The Planner uses node metadata to identify possible paths.

Search inputs:

- intent
    
- domain
    
- keywords
    
- previous task state
    

The Graph Engine confirms:

- dependencies
    
- valid paths
    
- required nodes
    

---

# 8. Ambiguous Requests

When user input is ambiguous:

Example:

```
Check my pipe
```

Possible workflows:

```
1. Thickness verification

2. Pressure test

3. Corrosion assessment
```

The Planner presents options.

The user selects the intended path.

---

# 9. Multi-standard Routing

The Planner supports multiple engineering standards.

Example:

```
API 570

ASME B31.3

BPVC Section VIII
```

When multiple standards apply:

The Planner:

1. identifies candidates
    
2. asks confirmation when required
    
3. records selected path
    

---

# 10. Planning Output Model

The Planner generates a detailed plan object.

Example:

```yaml
plan:

  goal:

    pipe integrity assessment


  intent:

    verification


  candidate_nodes:

    - integrity_check/root


  selected_nodes:

    - thickness_check


  missing_inputs:

    - design_temperature


  questions:

    - provide design temperature


  confidence:

    0.91
```

---

# 11. Missing Information Strategy

The Planner identifies minimum information required.

The Planner does not randomly request every input.

It analyzes dependencies.

Example:

If temperature is required for allowable stress:

Ask temperature before continuing.

The goal:

Collect only the information needed to unlock the next graph step.

---

# 12. User Direction Changes

If the user changes direction:

Example:

Current task:

```
Calculate thickness
```

User:

```
Actually perform pressure test verification
```

The Planner creates a new branch.

The existing task state is preserved.

The Task State Manager tracks:

- previous workflow
    
- abandoned path
    
- new request
    

---

# 13. Alternative Paths

The Planner supports alternative workflow suggestions.

Example:

```
Available approaches:

A:
Design verification

B:
Fitness assessment

C:
Inspection review
```

The user may select another path.

The selected path is stored in task state.

---

# 14. Memory Handling

The Planner does not permanently store conversation memory.

Conversation memory is managed by:

Task State Manager

The Planner retrieves:

- current task state
    
- previous decisions
    
- collected inputs
    
- conversation history
    

---

# 15. Planner and Task State Manager

The Planner can:

Read:

- conversation state
    
- task status
    
- previous decisions
    

Update:

- current intent
    
- selected workflow
    
- unresolved questions
    

The Task State Manager remains the owner of persistence.

---

# 16. Confidence Handling

Every Planner decision contains confidence.

Example:

```yaml

0.87
```

Confidence is used for:

- ambiguity detection
    
- asking confirmation
    
- selecting alternatives

# 17. Planner Failure Handling

If intent cannot be identified:

The Planner searches broadly.

Example:

```
1. Thickness calculation

2. Integrity verification

3. Pressure test

Which one should I continue with?
```

# 24. Final Architecture Principle

The Planner acts as the navigation intelligence layer.

System model:

```
Planner:

Decides where to look


Graph:

Defines dependencies


Rules:

Define allowed operations


Calculations:

Produce results


Reports:

Explain results
```

Final rule:

> The Planner may suggest the path, but the graph and rules determine the engineering outcome.

## Python implementation

> **Status:** Implemented in `engine/planner/`. `PlannerAgent` in `ai/agents/` is a thin adapter with LLM fallback.

### Pipeline

```
IntentResult (from IntentAgent)
        ↓
Planner.plan()
        ↓
GraphEngine.discover_roots() + build_plan preview
        ↓
NavigationPlan (missing inputs, selected root, questions)
        ↓
InputAgent (user-facing questions)
```

### Modules

| Module | Role |
|--------|------|
| `engine/planner/planner.py` | `Planner` — navigation coordinator |
| `engine/planner/tools.py` | `GraphTools`, `StateTools`, `RuleTools` |
| `models/planning.py` | `NavigationPlan`, `WorkflowCandidate`, `StructuredIntent` |
| `ai/agents/planner_agent.py` | Delegates to `Planner`; LLM fallback on ambiguity |
| `ai/prompts/planner/*.md` | Focused prompt stubs (§19) |

### Scope notes

- **pipe_wall_thickness_design** fully supported with graph-derived missing inputs (`material`, `design_temperature`, etc.).
- Ambiguous requests return `CLARIFY` with alternative workflow stubs.
- Planner decisions logged as `EventType.PLANNER_DECISION`; stored in `task.outputs.planning_summary`.
- No formula execution; graph execution deferred to `Executor` (STEP 4).
