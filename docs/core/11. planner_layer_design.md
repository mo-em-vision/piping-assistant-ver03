# Planner Layer Design

## Purpose

The Planner selects/ranks workflow candidates and orders the **next graph-active unresolved requirement** in phase order. It does not determine active engineering branches, invent dependencies, or author user-facing prompt copy.

Intent classification is owned by **`IntentAgent`** (`ai/agents/intent_agent.py`) upstream.

---

# Terminology

| Term | Meaning |
| --- | --- |
| **Planner Layer** | `engine/planner/` — navigation order, phases, canonical `EngineeringPlan`, legacy projections |
| **EngineeringPlan** | `models/engineering_plan.py` — **canonical** stored planner output (`requirements`, `phases`, `input_strategy`, `graph`, `traversal`) |
| **Planner** | `engine/planner/planner.py` — deterministic; builds/refreshes `EngineeringPlan`; returns legacy `NavigationPlan` projection |
| **PlannerAgent** | `ai/agents/planner_agent.py` — delegates to `Planner`; LLM fallback on ambiguity only |
| **NavigationPlan** | `models/planning.py` — **legacy read model** projected from `EngineeringPlan` (or phased navigation on unsupported paths) |
| **Graph Engine** | Expansion, `required_user_inputs()`, branch activation |

# Canonical output (`EngineeringPlan`)

For supported workflows (`pipe_wall_thickness_design`, `mawp_design`), the planner **builds and persists** a normalized `EngineeringPlan` on `task.outputs["engineering_plan"]` via `build_engineering_plan()` → `store_engineering_plan_on_task()`.

| Section | Authority for |
| --- | --- |
| `requirements` | Active `REQ-*` nodes, activation status, `depends_on`, question specs |
| `input_strategy` | `next_fields`, `submittable_fields`, `current_phase`, `blocked_fields` |
| `phases` | Phase ids, statuses, ordered `requirement_ids` |
| `graph` | Selected subgraph, branch decisions, expanded node ids |
| `traversal` | `PlannerTraversalState` — inspector/debug walk only |
| `root_goal` | Workflow completion anchor and blocking sets |

**Consumers must read planner navigation from `EngineeringPlan` (or projections derived from it), not from ad-hoc field lists or goal-store iteration.**

Ask order within a phase: graph expansion order + requirement `depends_on` depth, with generic tie-breakers on graph-derived metadata (`engine/planner/requirement_ordering.py`). Policy: [`docs/rules.md`](../rules.md) §13 (**Planner hardcode policy**).

Validation: `validate_engineering_plan()` after `finalize_engineering_plan()`. See [`docs/rules.md`](../rules.md) §20.

# Legacy and fallback projections

Use this table when wiring new features or debugging older paths. **Do not treat legacy shapes as sources of truth when `engineering_plan` is present.**

| Artifact | Location / builder | Role | Status | Prefer when |
| --- | --- | --- | --- | --- |
| **`EngineeringPlan`** | `task.outputs["engineering_plan"]`; `engineering_plan_for_task()` | Canonical stored plan | **Primary** | Always for supported workflows |
| **`input_strategy`** | On `EngineeringPlan` | Next gatherable field(s), submittable set, phase | **Primary** | API submission gate, timeline active row, composer |
| **`NavigationPlan`** | `navigation_plan_from_engineering_plan()`; `Planner.plan()` return | Legacy agent/chat read model | Projection | `PlannerAgent`, early clarify paths without stored plan |
| **`planning_projection()`** | `engine/state/goal_projection.py` | Planning-compatible dict for serializers | Derived | `task_state` / timeline; embeds plan when stored |
| **`goal_summary_dict()`** | `goal_projection.py` | Summary fields (`phase_missing`, `action`, …) | Derived | Inside `planning_projection`; prefers plan over goal store |
| **`goal_store`** | `task.execution_context.goal_store` | CLI / legacy goal tree | Derived sync | `apply_engineering_plan_to_goal_store()` on full refresh; **not** rewritten on `Planner.plan()` fast path (`skip_goal_migration=True`) |
| **`legacy_goal_map`** | Embedded on `EngineeringPlan`; `legacy_goal_map_for_task()` | Flat `GOAL-*` / `REQ-*` debug map | Deprecated | Inspector / backward compat only — not canonical |
| **`planning_summary`** | `task.outputs` (historical) | Pre–goal-store dict | Deprecated | Migrated once via `goals_from_planning_summary()` |
| **`engineering_plan_view`** | `build_engineering_plan_view()` | UI-friendly plan overview | Derived | Desktop inspector, timeline labels |
| **`planner_inspector_summary`** | `build_planner_inspector_summary()` | Compact planner debug payload | Rebuilt | Inspection API; always from stored plan when possible |
| **`planner_debug_projection`** | `build_planner_debug_projection()` | Dev Mode Planner tab contract | Derived | Inspector only — never drives execution |
| **`_legacy_navigation_plan()`** | `engine/planner/planner.py` | Direct `PhasedNavigation` → `NavigationPlan` | Fallback | Unsupported workflow or when engineering-plan refresh does not run |
| **Goal-store `NavigationPlan`** | `api/flow_guidance.navigation_plan_from_task()` | `planning_projection` → `NavigationPlan` | Fallback | No stored `engineering_plan` on task |

**Rule:** New code must not add features that read `goal_store` or flat `GOAL-*` maps when `engineering_plan` exists. Extend `EngineeringPlan` builders, validators, or projections instead.

# Owns / must not own

| Owns | Must not |
| --- | --- |
| Workflow candidate ranking from `IntentResult` | Active branches or parameter asks (graph expansion) |
| `EngineeringPlan` requirements, phases, `input_strategy` | Prompt copy (`engine/messaging/`) |
| `NavigationPlan` projection for agents and guidance | Formulas, compliance, calculations |
| Phase ordering enums and graph-derived requirement order | Per-field priority maps or workflow-specific field sequences (§13) |
| Goal-store sync from plan (`legacy_goal_adapter`) | Static lists of all branch parameters |

# Phased navigation

`NavigationPhase` order (see [`5. workflow_design.md`](5.%20workflow_design.md) §2):

`expansion_assumptions` → `path_decisions` → `parameter_gathering` → `coefficient_resolution` → `execution_assumptions` → `definition_equation_completion` → `ready`

`build_workflow_phased_navigation()` / `build_plan_phases_and_strategy()` partition missing fields from graph expansion — not a flat list of all possible inputs. Phases with no active requirements are omitted from `EngineeringPlan.phases`.

**Example (pipe wall thickness):** after expansion gates resolve, `input_strategy.next_fields` might advance to `design_temperature` before `internal_design_gage_pressure` (graph expansion order). Lookup outputs (S, Y, E, W) are derived requirements, not user-confirmed gathers.

Workflow reference: [`workflows/pipe-wall-thickness.yaml`](../../workflows/pipe-wall-thickness.yaml). Acceptance: [`docs/workflows/pipe_wall_thickness/acceptance_contract.md`](../workflows/pipe_wall_thickness/acceptance_contract.md).

# Prompt vs navigation

| Layer | Location |
| --- | --- |
| Which field, which phase | `EngineeringPlan.input_strategy` (canonical); `NavigationPlan` projection for legacy callers |
| Wording | `engine/messaging/` (`parameter_input_prompt`, `step_prompt`, `formula_parameter_prompt`) |
| Traversal narration | `GuidanceResolver` — not Planner |

Do not store new prompt strings on `NavigationPlan.questions` / `phase_questions` for greenfield features.

Policy: [`docs/rules.md`](../rules.md) §12, §13, §19.

# Final principle

The Planner behaves like a senior engineer deciding **where to look next** in the graph — not what the engineering answer is. The **`EngineeringPlan`** is the durable record of that decision; legacy shapes are projections for compatibility only.
