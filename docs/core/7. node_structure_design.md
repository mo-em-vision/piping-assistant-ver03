# Node Structure Design Documentation

## Purpose

This document defines the structure and behavior of engineering
knowledge graph nodes.

Nodes are deterministic engineering knowledge objects.

Core principle:

> Agents decide what to do next, but never what the truth is.

------------------------------------------------------------------------

# 1. Node Philosophy

A node represents a standard knowledge unit.

A node may contain:

-   paragraph text
-   formulas
-   conditions
-   requirements
-   limitations
-   assumptions (including `required_for_expansion`)
-   interactions (path decisions and value confirmations)
-   nomenclature (definition nodes; seeds parameter registry)
-   subsections and provisional_assumptions
-   references
-   notes

The design uses a hybrid structure:

-   paragraph traceability is preserved
-   formulas and execution logic remain modular



------------------------------------------------------------------------

# 2. Folder Structure

Example:

    standards/

    asme_b31.3/

    nodes/

    B313-304.1.1/

        node.md

        formulas/

        conditions/

        notes/

        references/

The standard is defined by the parent folder.

Nodes do not contain standard identifiers.

------------------------------------------------------------------------

# 3. Node Types

Every node has an explicit type.

Examples:

-   root
-   calculation
-   lookup
-   condition
-   requirement
-   definition
-   warning
-   report

------------------------------------------------------------------------

# 4. Node File Format

Each node uses markdown with YAML frontmatter.

Example:

    node.md

    ---
    metadata
    ---

    paragraph content

The full paragraph text is stored inside the node.

------------------------------------------------------------------------

# 5. Metadata Example

``` yaml
id:

B313-304.1.1

type:

calculation

version:

1.0

revision_year:

2024
```

`revision_year` records the published standard edition the node text is based on (shown in reference tabs). `version` remains the node implementation version for execution and reports.

------------------------------------------------------------------------

# 6. Dependencies

Execution order is created only from dependencies.

Example:

``` yaml
depends_on:

- B313-material-stress

- TABLE-A1
```

The Graph Engine determines traversal and execution order.

------------------------------------------------------------------------

# 7. Inputs

Inputs define:

-   required values
-   units
-   source
-   limitations

Example:

``` yaml
inputs:

- name:

    design_pressure

  unit:

    Pa

  source:

    user_input
```

------------------------------------------------------------------------

# 8. Input Traceability

Every value keeps its origin.

Possible sources:

-   user input
-   another node
-   table lookup
-   default value

Example:

``` yaml
source:

node:B313-314.5.4
```

------------------------------------------------------------------------

# 9. Unit Management

A global unit conversion function converts all calculation values to SI
units.

The original value is preserved.

Example:

User:

500 psi

Stored:

original: 500 psi

calculation: 3447378 Pa

------------------------------------------------------------------------

# 10. Formula Model

Formulas are stored separately.

Example:

    formulas/

    wall_thickness.md

Formula contains:

-   equation
-   variables
-   steps
-   execution function

Example:

``` yaml
formula:

name:

wall_thickness


executor:

calculate_wall_thickness()
```

------------------------------------------------------------------------

# 11. Formula Execution

Formulas may call deterministic functions.

Functions:

-   execute calculations
-   validate units
-   return intermediate values

Functions do not interpret standards.

------------------------------------------------------------------------

# 12. Conditions

Conditions are stored inside nodes.

Example:

``` yaml
conditions:

- name:

    thin_wall_check

  expression:

    t < D/6
```

------------------------------------------------------------------------

# 13. Tables and Lookups

Tables are resolved through the Reference Resolver.

Example:

``` yaml
table:

A1
```

Lookup rules can be stored:

-   lower value selection
-   no interpolation
-   applicability limits

------------------------------------------------------------------------

# 14. Defaults

Nodes support defaults.

Example:

``` yaml
default:

1.0

requires_confirmation:

true
```

Defaults require user confirmation.

------------------------------------------------------------------------

# 15. Limitations

Limitations should be structured.

Example:

``` yaml
limitations:

- parameter:

    temperature

  action:

    reject
```

------------------------------------------------------------------------

# 16. Report Requirements

Nodes define report output requirements.

Example:

``` yaml
report:

include:

- inputs

- formulas

- decisions

- warnings
```

------------------------------------------------------------------------

# 17. AI Hints

Nodes may contain optional AI hints.

Example:

``` yaml
ai_hints:

explain:

temperature limitation clearly
```

AI hints only affect explanation.

They never affect calculation.

------------------------------------------------------------------------

# 18. Traceability

Execution stores:

-   node ID
-   paragraph reference
-   formula
-   inputs
-   decisions
-   intermediate values
-   outputs
-   warnings

Intermediate values are not rounded.

------------------------------------------------------------------------

# 19. Sample ASME B31.3 Wall Thickness Node

> **Status:** Implemented at `standards/asme_b31.3/nodes/B313-304.1.1/` (definition) and `B313-304.1.2/` (calculation).  
> Root entry: `standards/asme_b31.3/roots/pipe_wall_thickness_design/root.md`.

Structure:

```
standards/asme_b31.3/nodes/B313-304.1.1/

    node.md                    # definition: nomenclature, interactions, subsections

    equations/

        eq_2_minimum_required_thickness.md

standards/asme_b31.3/nodes/B313-304.1.2/

    node.md                    # calculation: inputs, provisional_assumptions, subsections (a)/(b)

    equations/

        wall_thickness.md

    conditions/

        thin_wall_check.md

    notes/

        corrosion_and_ordered_thickness.md
```

Display formula (§304.1.2 thin-wall, eq. 3a):

```
t = PD / 2(SEW + PY)
```

Minimum ordered thickness (§304.1.1 eq. 2):

```
t_m = t + c
```

The sample nodes represent:

-   **304.1.1** — definition node: nomenclature with `introduced_here` / `resolution`, expansion interactions (`straight_pipe_section`, `pressure_loading`), subsection and eq-2 reference
-   **304.1.2** — calculation node: required inputs, `joint_category` confirmation, provisional `thin_wall`, dependency on `B313-302.3.5` for W, thin-wall check condition
-   dependencies (`B313-material-stress`, `B313-302.3.5`)
-   formula execution (`equations/wall_thickness.md`, `calculate_minimum_required_thickness` for t_m)
-   validation requirements (`conditions/thin_wall_check.md`)
-   report trace metadata in `node.md` frontmatter

## Interactions → CLI numbered options

Node `interactions` and expansion `assumptions` with `options` / yes-no gates drive **deterministic** user prompts — not LLM text generation.

| Source | CLI behavior |
|--------|----------------|
| `assumptions` with `allowed_values: [true, false]` | Numbered yes/no (e.g. `straight_pipe_section`) |
| `interactions` with `mode: decision` and `options` | Numbered path choices (e.g. `pressure_loading`, `joint_category`, `d_input_mode`) |
| `inputs` with `requires_confirmation: true` and `default` | Numbered confirm-default vs enter-value (e.g. E, W, Y) |

Parsed by `engine/graph/node_interaction.py` and formatted by `engine/messaging/step_prompt.py` / `formula_parameter_prompt.py`.

------------------------------------------------------------------------

# 20. Design Principle

Nodes are flexible engineering knowledge objects.

They provide:

-   deterministic execution
-   auditability
-   extensibility across standards
-   reproducible calculations
