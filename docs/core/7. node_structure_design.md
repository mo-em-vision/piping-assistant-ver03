# Node Structure Design Documentation

## Purpose

This document defines the structure and behavior of engineering
knowledge graph nodes.

Nodes are deterministic engineering knowledge objects.

Core principle:

> Agents decide what to do next, but never what the truth is.

------------------------------------------------------------------------

# 1. Node Philosophy

A node represents a standard knowledge unit.

A node may contain:

-   paragraph text
-   formulas
-   conditions
-   requirements
-   limitations
-   assumptions (including `required_for_expansion`)
-   interactions (path decisions and value confirmations)
-   nomenclature (definition nodes; seeds parameter registry)
-   subsections and provisional_assumptions
-   references
-   notes

The design uses a hybrid structure:

-   paragraph traceability is preserved
-   formulas and execution logic remain modular



------------------------------------------------------------------------

# 2. Folder Structure

Example (ASME B31.3 — flat pack layout):

    standards/asme/asme_b31.3/nodes/

    B313-304.1.1/
        node.yaml              # structure, nomenclature, embedded assets (source: blocks), paragraph body

    304.1.2-a/
        node.yaml              # equations, conditions, notes inlined as source: metadata

    302.3.5-e/
        node.yaml
        eq_1a_*.py             # execution modules only (siblings of node.yaml; no subfolders)

    B313-WF-PIPE-WALL-THICKNESS/
        node.yaml

    B313-param-P/
        node.yaml

    B313-interaction-pressure-loading/
        node.yaml

Each node folder contains **files at one level only**. Asset content (equations, conditions, notes, references) lives as embedded `source:` blocks inside `node.yaml`, not under `equations/`, `conditions/`, etc. Cross-node equation references use graph paths or embedded aliases (e.g. `nodes/304.1.2-a/equations/mawp_pressure.md`) resolved via `StandardsReader.read_asset_text`. The only exception is co-located `.py` calculation modules at the node root (B31.3 §302.3.5).

The standard is defined by the parent pack folder. B31.3 uses a **flat** layout: `nodes/{node_id}/` (node id matches folder name). Other packs may use the same pattern. Section grouping for browse/navigation comes from `index.md`, not folder depth.

Nodes do not repeat the standard name in their metadata.

------------------------------------------------------------------------

# 3. Node Types

Every node has an explicit type.

**Micro-graph canonical types** (compiled into `PackGraph`):

-   `workflow` — root orchestration
-   `definition` — standard paragraph anchor; nomenclature, expansion gates, child references
-   `calculation` — executable section; inputs, equations, conditions
-   `equation` — sympy calculations and lookups (`kind: calculation` or `lookup`)
-   `parameter` — task-state binding (`kind: assumption`, `interaction`, or unset)
-   `quantity`, `designation` — shared engineering concepts
-   `text` — initiation text, equation intros, notes (`kind: section`, `table`, `note`, …)
-   `unit` — global unit ontology

Legacy on-disk types (`assumption`, `interaction`, `lookup`, `table`, `standard_section`) normalize at compile time via `engine/reference/node_types.py`.

------------------------------------------------------------------------

# 4. Node File Format

Section nodes use a **single `node.yaml`** when structure and paragraph traceability are combined (merged from the former dual-file pattern):

| Content | Location |
|---------|----------|
| Graph structure | YAML frontmatter: `type`, `contains`, `requires`, `assumptions`, `interactions`, `equations`, `texts` |
| Nomenclature & paragraph trace | YAML frontmatter + markdown body after the closing `---` |

Legacy nodes may still use `node.md` only (tables, notes). New section nodes should use one `node.yaml` per folder.

When both `node.yaml` and `node.md` exist (pre-merge), **YAML is preferred** for graph discovery; loaders merge markdown-only keys at runtime until consolidated with `scripts/merge_b313_node_sources.py`.

Standalone nodes (workflows, parameters, equations in their own folder) use a single `node.yaml` (or legacy `node.md`).

Example `node.yaml`:

    ---
    id: B313-304.1.1
    type: definition
    contains:
      - B313-eq-2
      - B313-assumption-straight-pipe
    assumptions:
      - id: B313-assumption-straight-pipe
        type: parameter
        kind: assumption
        input_id: straight_pipe_section
        ...
    ---

    # ASME B31.3 §304.1.1

The full paragraph text lives in `node.md` below its frontmatter (or in the YAML body when no separate markdown file is used).

# 5. Metadata Example

``` yaml
id: B313-304.1.1

type: definition

version: "1.0"

revision_year: 2024
```

`revision_year` records the published standard edition the node text is based on (shown in reference tabs). `version` remains the node implementation version for execution and reports.

------------------------------------------------------------------------

# 6. Dependencies

Execution order is created only from dependencies.

Example:

``` yaml
depends_on:

- B313-material-stress

- TABLE-A1
```

The Graph Engine determines traversal and execution order.

------------------------------------------------------------------------

# 7. Inputs

Inputs define:

-   required values
-   units
-   source
-   limitations

Example:

``` yaml
inputs:

- name:

    design_pressure

  unit:

    Pa

  source:

    user_input
```

------------------------------------------------------------------------

# 8. Input Traceability

Every value keeps its origin.

Possible sources:

-   user input
-   another node
-   table lookup
-   default value

Example:

``` yaml
source:

node:B313-314.5.4
```

------------------------------------------------------------------------

# 9. Unit Management

A global unit conversion function converts all calculation values to SI
units.

The original value is preserved.

Example:

User:

500 psi

Stored:

original: 500 psi

calculation: 3447378 Pa

------------------------------------------------------------------------

# 10. Formula Model

Equations are referenced from parent section metadata under an `equations:` list. Each entry may:

-   point to a separate file via `file:` (legacy path, kept as alias), and/or
-   embed the full equation definition inline via a `source:` block (YAML frontmatter + markdown body)

Embedded `source:` is the preferred authoring form for migrated B31.3 nodes. `GraphBuilder` and `StandardsReader` extract embedded children as first-class graph nodes via `engine/reference/embedded_nodes.py`.

Example (embedded in parent `node.md` or `node.yaml`):

```yaml
equations:
  - id: B313-eq-wall-thickness
    type: equation
    file: equations/wall_thickness.md
    source: |
      ---
      equation_id: wall_thickness
      sympy: "t = P*D / (2*(S*E*W + P*Y))"
      display_latex: "t = PD / 2(SEW + PY)"
      requires:
        - node_id: B313-quantity-pressure
          alias: P
          priority: 40
      calculates:
        - B313-param-t
      ---
      # Internal Pressure Wall Thickness
```

Equation metadata contains:

-   `sympy` / `display_latex` — display and evaluation
-   `requires` / `calculates` — dataflow
-   `variables`, `steps`, `executor` — legacy calculation-engine path (where still used)

The same embedded pattern applies to `notes`, `texts`, `conditions`, `assumptions`, and `interactions` containers on parent nodes.

------------------------------------------------------------------------

# 11. Formula Execution

Formulas may call deterministic functions.

Functions:

-   execute calculations
-   validate units
-   return intermediate values

Functions do not interpret standards.

------------------------------------------------------------------------

# 12. Conditions

Conditions are stored inside nodes.

Example:

``` yaml
conditions:

- name:

    thin_wall_check

  expression:

    t < D/6
```

------------------------------------------------------------------------

# 13. Tables and Lookups

Tables are resolved through the Reference Resolver.

Example:

``` yaml
table:

A1
```

Lookup rules can be stored:

-   lower value selection
-   no interpolation
-   applicability limits

------------------------------------------------------------------------

# 14. Defaults

Nodes support defaults.

Example:

``` yaml
default:

1.0

requires_confirmation:

true
```

Defaults require user confirmation.

------------------------------------------------------------------------

# 15. Limitations

Limitations should be structured.

Example:

``` yaml
limitations:

- parameter:

    temperature

  action:

    reject
```

------------------------------------------------------------------------

# 16. Report Requirements

Nodes define report output requirements.

Example:

``` yaml
report:

include:

- inputs

- formulas

- decisions

- warnings
```

------------------------------------------------------------------------

# 17. AI Hints

Nodes may contain optional AI hints.

Example:

``` yaml
ai_hints:

explain:

temperature limitation clearly
```

AI hints only affect explanation.

They never affect calculation.

------------------------------------------------------------------------

# 18. Traceability

Execution stores:

-   node ID
-   paragraph reference
-   formula
-   inputs
-   decisions
-   intermediate values
-   outputs
-   warnings

Intermediate values are not rounded.

------------------------------------------------------------------------

# 19. Sample ASME B31.3 Wall Thickness Node

> **Status:** Implemented at `standards/asme/asme_b31.3/nodes/B313-304.1.1/` (`type: definition`) and `nodes/304.1.2-a/` (`type: calculation`).  
> Root entry: `standards/tasks/asme_b31.3/pipe_wall_thickness_design/root.md`.

Structure:

```
standards/asme/asme_b31.3/nodes/B313-304.1.1/
    node.yaml                  # definition: contains, assumptions, interactions, equations, nomenclature, paragraph body

standards/asme/asme_b31.3/nodes/304.1.2-a/
    node.yaml                  # calculation: requires, calculates, contains, paragraph body
    equations/                 # legacy file: aliases only — prefer embedded source: in node.yaml
```

Child nodes (`B313-eq-2`, `B313-assumption-straight-pipe`, `B313-eq-wall-thickness`, etc.) are **embedded** in the parent `node.yaml` / `node.md` metadata containers. They compile as first-class graph nodes without requiring separate per-child folders. Legacy `equations/*.md` files remain as `file:` aliases.

Display formula (§304.1.2 thin-wall, eq. 3a):

```
t = PD / 2(SEW + PY)
```

Minimum ordered thickness (§304.1.1 eq. 2):

```
t_m = t + c
```

The sample nodes represent:

-   **304.1.1** (`type: definition`) — nomenclature with `introduced_here` / `resolution`; expansion gates in `assumptions` / `interactions` (`straight_pipe_section`, `pressure_loading`); eq-2 and initiation text embedded under `equations` / `texts`
-   **304.1.2** (`type: calculation`) — required parameter refs, provisional `thin_wall`, dependency on `302.3.5-e` for W; thin-wall condition, wall-thickness equation, and notes embedded in `node.yaml` (legacy `node.md` / `equations/` aliases may remain)
-   dependencies (`B313-material-stress`, `302.3.5-e`)
-   equation execution via embedded `B313-eq-wall-thickness` (`sympy`) and legacy `calculate_minimum_required_thickness` for t_m where applicable
-   validation (`thin_wall_check` embedded under `conditions`)
-   report trace metadata in frontmatter

## Interactions → CLI numbered options

Node `interactions` and expansion `assumptions` with `options` / yes-no gates drive **deterministic** user prompts — not LLM text generation.

| Source | CLI behavior |
|--------|----------------|
| `assumptions` with `allowed_values: [true, false]` | Numbered yes/no (e.g. `straight_pipe_section`) |
| `interactions` with `mode: decision` and `options` | Numbered path choices (e.g. `pressure_loading`, `joint_category`, `d_input_mode`) |
| `inputs` with `requires_confirmation: true` and `default` | Numbered confirm-default vs enter-value (e.g. E, W, Y) |

Parsed by `engine/graph/node_interaction.py` and formatted by `engine/messaging/step_prompt.py` / `formula_parameter_prompt.py`.

# 20. Design Principle

Nodes are flexible engineering knowledge objects.

They provide:

-   deterministic execution
-   auditability
-   extensibility across standards
-   reproducible calculations
