# Node Structure Design

> **Design reference (current).** New YAML must follow [`audits/contracts/nodes/00-START-HERE.md`](../../audits/contracts/nodes/00-START-HERE.md) and per-type contracts — not this file alone.

## Purpose

This document describes the **current** micro-graph node model: how engineering knowledge is authored, compiled, and consumed at runtime.

Nodes are deterministic, immutable knowledge objects. Runtime values (Facts, goals, validation outcomes) live in the Execution Context — never in node files.

Core principle:

> Agents decide what to do next, but never what the truth is.

---

# 1. Node philosophy

Each node represents **one** independently meaningful engineering concept ([`Constitution.md`](../architecture/Constitution.md) Law 3).

| Concern | Where it lives |
| --- | --- |
| Official paragraph text, hierarchy, applicability | `paragraph` node |
| Gatherable engineering fields | `parameter` (`PARAM-*`) nodes |
| Deterministic formulas | `equation` nodes |
| Table resolution | `lookup` nodes |
| Compliance checks | `validation_rule` nodes |
| Engineering objectives and phase order hints | `workflow` nodes under `workflows/` |
| Tabular data | `table` nodes |
| Governing sources | `authority` nodes |

Paragraph nodes are **not** self-executing calculators. Formula execution, lookup resolution, and compliance checks belong on their respective node types.

---

# 2. Canonical node types

Authoring types are defined in [`audits/contracts/nodes/00-START-HERE.md`](../../audits/contracts/nodes/00-START-HERE.md):

`workflow`, `paragraph`, `parameter`, `equation`, `lookup`, `validation_rule`, `table`, `table_note`, `authority`, `concept`, `text`, `unit`, `dimension`, `quantity`, `designation`.

**Retired authoring types (do not use for new nodes):** `definition`, `calculation`, `root`, and the monolithic paragraph model that inlined formulas, lookups, and validation in one file.

Legacy names may appear only in labelled compatibility sections or in compile-time normalization (`engine/reference/node_types.py`) for old packs.

# 3. Pack layout

Standards packs live under `knowledge/standards/<publisher>/<pack>/`:

```
asme_b31.3/
├── pack.yaml
├── index.md
├── nodes/
│   ├── paragraph/       # {id}.yaml — one file per lettered subsection
│   ├── equation/
│   ├── parameter/
│   ├── lookup/
│   ├── validation_rule/
│   └── tables/
└── asme_b313_graph.db   # optional compile cache
```

User-intent entry points are **workflow** YAML files in repository [`workflows/`](../../workflows/) — not `roots/.../root.md` or `standards/tasks/...`.

Example workflow: [`workflows/pipe-wall-thickness.yaml`](../../workflows/pipe-wall-thickness.yaml) (`key: pipe_wall_thickness_design`).

Example paragraph: [`knowledge/standards/asme/asme_b31.3/nodes/paragraph/304.1.2-a.yaml`](../../knowledge/standards/asme/asme_b31.3/nodes/paragraph/304.1.2-a.yaml).

Example equation: [`knowledge/standards/asme/asme_b31.3/nodes/equation/asme-b313-304-1-2-eq-3a.yaml`](../../knowledge/standards/asme/asme_b31.3/nodes/equation/asme-b313-304-1-2-eq-3a.yaml).

# 4. Relationships

All outgoing relationships use typed `edges: [{type, target}]` per [`audits/contracts/nodes/01-shared-node-contract.md`](../../audits/contracts/nodes/01-shared-node-contract.md).

- Do **not** author top-level `depends_on`, `links`, `parent`, `child`, `next`, or `previous` on new nodes.
- Paragraph parent/child order uses `hierarchy` metadata — not structural edges.
- Cross-paragraph citations use `related_to` edges.

The Graph Engine determines traversal, branch activation, and execution order from compiled edges and expansion policy (`expand_workflow`, `engine/graph/expansion_policy.py`).

---

# 5. Paragraph nodes

Paragraph nodes hold:

- `text` (official prose; subsection labels may use `(a)` style in prose while `id` uses hyphen form, e.g. `304.1.2-a`)
- `paragraph_number`, `key`
- `hierarchy` (parent, ordered children)
- `metadata` (`last_revision`, `edited_by`, optional `kind: nomenclature`)
- contract-approved nested `execution` metadata (applicability, assumptions, interactions)
- `introduces_parameter` edges to `PARAM-*` nodes

Paragraph nodes do **not** own formula steps, lookup tables, or compliance logic. Nomenclature paragraphs introduce parameters via edges; tables belong on `table` / `lookup` nodes.

Contract: [`audits/contracts/nodes/paragraph.md`](../../audits/contracts/nodes/paragraph.md).

---

# 6. Equation nodes

Equation nodes own deterministic formula execution:

- `sympy` / `display_latex` (or equivalent contract fields)
- `requires_parameter` / `depends_on_equation` edges
- output parameter references

The Execution Layer evaluates equations; the Graph Engine only schedules them on the active path.

Contract: [`audits/contracts/nodes/equation.md`](../../audits/contracts/nodes/equation.md).

---

# 7. Lookup nodes

Lookup nodes resolve table-driven values (allowable stress, joint efficiency E, W, Y, outside diameter from NPS, etc.).

- Users provide **lookup keys** or select an authored resolution branch — not lookup outputs.
- Resolution strategy (exact match, interpolation, conservative selection, etc.) is defined by the lookup contract/rule — not by a global "always select the lower value" rule.
- Do **not** author `default`, `requires_confirmation`, or user-confirmed defaults for lookup-derived coefficients.

Contracts: [`audits/contracts/nodes/lookup.md`](../../audits/contracts/nodes/lookup.md), [`docs/rules.md`](../rules.md) §16.

---

# 8. Validation-rule nodes

Compliance checks (temperature limits, applicability, pass/fail rules) belong on `validation_rule` nodes.

`ValidationEngine` evaluates compliance at plan and per-node boundaries. `RuleEngine` handles only narrow runtime expression conditions delegated during node execution — not the full compliance gate.

Contract: [`audits/contracts/nodes/validation-rule.md`](../../audits/contracts/nodes/validation-rule.md).

---

# 9. Parameter nodes

`PARAM-*` nodes define gatherable engineering concepts (symbol, dimension, composer metadata). Values are stored as runtime Facts — never inside parameter nodes.

User-supplied parameters use `metadata.composer_input` and related fields per the parameter contract.

Contract: [`audits/contracts/nodes/parameter.md`](../../audits/contracts/nodes/parameter.md).

---

# 10. Workflow nodes

Workflow YAML under `workflows/` declares:

- engineering objective (`goal_expansion`, `entry_points`)
- phase ordering hints (`runtime.navigation` — order and gate fields only)
- gate interactions (`runtime.interactions`, `runtime.navigation.assumption_gate_fields`)

Graph expansion determines **which** parameters and branches are active; workflow nested navigation supplies **order** only. See [`docs/rules.md`](../rules.md) §13.

Contract: [`audits/contracts/nodes/workflow.md`](../../audits/contracts/nodes/workflow.md).

---

# 11. Compilation and runtime

```
Markdown/YAML nodes → GraphBuilder → PackGraph → optional *_graph.db cache
```

- Source of truth: `knowledge/` and `workflows/`
- Embedded children in metadata containers compile as first-class nodes via `engine/reference/embedded_nodes.py`
- Runtime reads the graph; it never writes back to knowledge files

# 12. Sample — pipe wall thickness (current paths)

| Role | Current path |
| --- | --- |
| Workflow entry | [`workflows/pipe-wall-thickness.yaml`](../../workflows/pipe-wall-thickness.yaml) |
| Nomenclature / expansion gates | `knowledge/standards/asme/asme_b31.3/nodes/paragraph/304.1.1-b.yaml` (and related §304.1.1 subsections) |
| Thin-wall equation | `knowledge/standards/asme/asme_b31.3/nodes/equation/asme-b313-304-1-2-eq-3a.yaml` |
| Allowable stress lookup | lookup nodes under the B31.3 pack |
| Pipe dimensions | `knowledge/standards/asme/asme_b36.10/` tables + lookup nodes |

Display formula (eq. 3a): `t = PD / 2(SEW + PY)`.

Expansion assumptions (`straight_pipe_section`), path decisions (`pressure_design_case`), and parameter gathering are driven by graph expansion and planner phase order — not by hardcoded field lists in application code.

---

# 13. User prompts and interactions

Gate and branch choices come from authored `assumptions`, `interactions`, and workflow `runtime.interactions`. Deterministic prompt copy is owned by `engine/messaging/` (`step_prompt.py`, `formula_parameter_prompt.py`). Traversal narration is owned by `GuidanceResolver` (`docs/rules.md` §21).

Parsed by `engine/graph/node_interaction.py` and formatted by messaging builders — not by LLM agents during active deterministic phases.

---

# 14. Traceability

Execution stores per-node trace: node id, paragraph reference, formula, inputs, decisions, intermediate values, outputs, warnings. Intermediate values are not rounded until final presentation.

# Legacy compatibility

The following existed in earlier designs and may still resolve at runtime for old sessions or tests — **do not author new content this way**:

| Legacy pattern | Current replacement |
| --- | --- |
| `type: definition` / `type: calculation` paragraph bundles | Separate `paragraph`, `equation`, `lookup`, `validation_rule` nodes |
| `roots/.../root.md` workflow entry | `workflows/{machine-key}.yaml` |
| `standards/tasks/...` task roots | `workflows/*.yaml` |
| Top-level `standards/...` pack tree | `knowledge/standards/...` |
| Per-node folder trees with `node.yaml` + `equations/` subfolders | Flat `nodes/{type}/{id}.yaml` under pack |
| Top-level `depends_on` / `links` blocks | Typed `edges` per node contract |
| `integrity_check/root`, `wall_thickness/root` slug examples | Workflow `key` / `id` (e.g. `pipe_wall_thickness_design`, `WF-PIPE-WALL-THICKNESS`) |

`normalize_root_id()` in `engine/graph/graph_engine.py` maps historical `roots/.../root.md` strings to workflow slugs for read-only compatibility.

---

# Design principle

Nodes provide deterministic, auditable, reusable engineering knowledge. The graph defines what must happen on the active path; validation defines what is allowed; execution produces Facts; reports and presentation explain what happened — without modifying engineering truth.
