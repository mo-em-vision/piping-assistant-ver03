# Component Responsibilities Documentation

## Purpose

This document defines the responsibility boundaries between system components.

The architecture separates:

- engineering truth
- validation
- navigation
- execution
- state management
- audit
- reporting


Core principle:

> Agents decide what to do next, but never what the truth is.

# Layer 1 — Deterministic Graph Engine

## Purpose

The Graph Engine is the engineering truth layer.

It defines:

- what depends on what
- execution order
- dependency traversal
- graph versions


## Responsibilities

Handles:

- standard node loading
- graph creation
- dependency resolution
- DAG traversal
- topological sorting
- graph version persistence
- execution planning


## Inputs

Receives:

- user intent result
- selected root node
- standard nodes


## Outputs

Produces:

- discovered node set
- dependency graph
- execution order


## Restrictions

The Graph Engine must not:

- interpret user language
- modify formulas
- validate engineering constraints
- perform calculations

# Layer 2 — Rule Engine

## Purpose

The Rule Engine determines:

"is this allowed?"


## Responsibilities

Handles:

- unit validation
- input constraints

- temperature limits
- assumptions

- table applicability
- warnings


## Flow

# Calculation Engine

## Purpose

Executes deterministic engineering calculations.


## Responsibilities

Handles:

- formula execution
- calculation steps
- intermediate values
- final results


## Requirements

Must preserve:

- intermediate values
- units
- calculation trace


Intermediate values are not rounded.

Rounding occurs only during final presentation.


---
## What a Node Is

A node represents one ASME B31.3 paragraph or logical unit.

Each node is **self-contained** and **self-executing**. It carries:

- The full verbatim text of the paragraph (so the LLM never needs an external source)
- Structured metadata in YAML frontmatter (so the dependency resolver can parse it without reading prose)
- Its position in the hierarchy (parent, children, cross-references)
- Its applicability conditions (checked before the node runs)
- Its inputs and their sources (user, another node, or a confirmed default)
- Its formula as an ordered list of named expression steps (evaluated by the generic executor)
- Its constants (fixed values the node owns — no external lookup needed)
- Its post-execution checks (validations and re-routing logic)
- Its outputs (what is stored and passed to dependent nodes)
- Its dependencies on other nodes (for topological sort)
- Its report metadata (how its result appears in the grouped output)

# Agent Layer

## Purpose

Agents provide navigation intelligence.


Agents handle:

- intent detection
- node selection strategy
- missing information resolution
- multi-standard routing
- explanation


Agents do not:

- calculate
- choose formulas
- override rules
- modify outputs


Principle:

> Agents decide what to do next, but never what the truth is.


---

# Task Continuity and Context Handling

A session may contain multiple tasks.


The system tracks:

- active task
- previous calculations
- inputs
- assumptions


If user provides unrelated input:


Example:

Current:

	pipe thickness calculation


User:

	weather question


The system:


- detects context mismatch
- preserves engineering task
- asks whether to continue or start another task


The task is never silently abandoned.


---

# Input Change Detection

If inputs change:


Previous:

	Temperature = 300 F


New:

	Temperature = 600 F


The system detects the conflict.


The user is warned:

	"The previous calculation may no longer be valid. A new full calculation is required."

# Final Architecture Principle

Graph defines what happens.

Rules define what is allowed.

Agents define where to look.
